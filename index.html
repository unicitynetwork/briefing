Repeatable Quest Schedules with Period-Bound Verification
PRs #6 (backoffice), #20 (API), and #6 (quest-frontend) together introduce repeatable quest scheduling for the ONCHAIN platform. The API now enforces period-bound verification so that quest completions are scoped to a defined time window rather than being globally unique. A hotfix in PR #19 immediately followed to reset stale userStatus records for any prior-period status, ensuring users who had a non-fresh status from a previous period are not incorrectly blocked. On the frontend, PR #6 delivers a single-click check-in flow alongside a cooldown UX that surfaces when a repeatable quest is in its locked/cooling-down state. Together these changes make recurring on-chain quests viable end-to-end, from schedule configuration through backend enforcement to user-facing feedback.
sphere-backoffice, sphere-api, sphere-quest-frontend
Misuse-Resistant Backend Auth via verifySphereAuth
PR #145 in sphere-sdk exposes two new public utilities: verifySphereAuth, a high-level helper for validating Sphere authentication tokens in backend services, and computeDirectAddressFromChainPubkey, which derives a deterministic address from a chain public key without requiring a live network call. The explicit goal is misuse-resistance — bundling the correct verification logic so integrators cannot easily implement it incorrectly. PR #3 in unicity-claude-marketplace follows up by updating the backend-auth tutorial in the sphere-connect documentation to use the new verifySphereAuth recipe as the canonical example, replacing any prior ad-hoc approach. This pairing of SDK surface and updated documentation ensures new integrators adopt the secure path by default.
sphere-sdk, unicity-claude-marketplace
NostrClient Deduplication and Transport Cleanup
PR #125 refactors the sphere-sdk transport layer to share a single NostrClient instance across both the transport and Mux components, eliminating the previously duplicated client instantiation. As a consequence, the redundant healthcheck that was tied to the second client instance has been removed, reducing unnecessary network chatter on SDK initialisation. This is a reliability and maintainability improvement: fewer client instances mean fewer potential desync issues between transport and multiplexer state. The change has no user-facing API impact but lowers the SDK's resource footprint in environments where many connections are managed concurrently.
sphere-sdk
