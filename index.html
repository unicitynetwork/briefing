Astrid17 PRs merged
AOS CE product identity and workspace foundation
PRs #1–#4 and #14 establish the Unicity AOS product workspace as a Cargo workspace, define the internal runtime control boundary, make the public surface product-only, and rename the product CLI to 'aos'. The .github PRs #2 and #3 set up and restore the AOS organization profile. Together these changes lay the structural and branding foundation for the Community Edition distribution.
aos-ce, .github
Unicity CE distribution selection and release boundary
PRs #10, #11, and #12 rename the Community Edition distro to 'Unicity CE', wire it as the default selection during product init, and establish a formal CE release boundary. This creates a clean, versioned separation between the CE product tier and any future editions, making release management explicit and auditable.
aos-ce
Runtime bootstrap, delegation, and legacy migration
PRs #6, #7, #8, and #9 introduce an AOS runtime bootstrap seam, harden home directory resolution during bootstrap, add Unicity runtime command delegation, and implement explicit legacy runtime migration. These changes ensure the new runtime layer can safely intercept and delegate commands while providing a controlled upgrade path for users on older runtimes.
aos-ce
Legacy handoff preservation and migration error reporting
PRs #15 and #16 build on the migration foundation by preserving legacy distro handoff paths and fixing error reporting when migration receipt handoff fails. Surfacing handoff errors explicitly prevents silent failures during runtime transitions and ensures operators have actionable feedback when a migration does not complete cleanly.
aos-ce

Pubkey-first identity display, hide DIRECT:// addresses
PR #431 removes DIRECT:// address strings from all UI surfaces and replaces them with a pubkey-first identity display model. This reduces exposure of internal routing addresses to end users and aligns the identity presentation with the network's cryptographic identity primitives rather than transport-layer details.
sphere
Sentry noise reduction and error grouping stability
PR #430 applies a second-generation Sentry filter that drops roughly 49% of remaining event volume attributed to browser extension interference. The release is pinned and error grouping rules are stabilized, which improves signal-to-noise ratio in the error dashboard and makes regression detection more reliable for the engineering team.
sphere
Wallet API observability: metrics, alerts, and gauges
PR #105 introduces a comprehensive observability layer to the wallet API, adding size histograms, rejection counters, a configuration gauge, and high-water gauges across key subsystems. Nine alert rules are defined alongside these metrics, giving operators threshold-based visibility into request sizing, rejection rates, and resource utilization without requiring manual log triage.

SMT switches to big-endian bit ordering
PR #170 changes the Sparse Merkle Tree implementation in the Go aggregator to use big-endian bit ordering. This alignment is typically required for interoperability with SMT implementations in other languages or protocol layers that assume network byte order. Consistent bit ordering across the stack prevents subtle proof verification failures when roots or proofs are exchanged between components.
