SSL Docker Deployment with HAProxy and CI Hardening
PRs #291, #298, #299, and #300 collectively deliver and harden a production-grade SSL Docker deployment for Sphere. #291 introduced the initial HAProxy integration with an SSL entrypoint, while #298 tightened the Docker entrypoint configuration and added CI validation for the SSL path. #299 re-enabled the Dockerfile.ssl target in the validate matrix after it was inadvertently dropped, and #300 completed a second hardening pass covering drain budget tuning, CI smoke tests, and trimming the SSL_REQUIRED logic. Together these changes give Sphere operators a robust, continuously validated SSL termination path through HAProxy.
unicity-sphere
Resilient Transport Layer with Retry and Health Checks
PRs #108 and #113 overhaul the SDK transport layer to handle unreliable relay connectivity. #108 introduced resilient event publishing, a health-check mechanism, and a shared address resolver to reduce redundant lookups and surface relay availability early. #113 added automatic retry logic for publishEvent calls that are rejected by the relay, preventing silent message loss under transient failures. These changes significantly improve the reliability of event delivery in degraded network conditions, which is critical for payment and swap flows that depend on relay liveness.
unicity-sphere, sphere-sdk
SpendQueue Race Condition Fixes in Concurrent Payments
PRs #87 and #107 address a TOCTOU (time-of-check-time-of-use) race condition in the payments module that could cause incorrect behavior during concurrent send() calls. #87 identified and eliminated the fundamental race by restructuring state checks and mutations to be atomic within the SpendQueue. #107 followed up with further fixes to the concurrent send path, ensuring the queue correctly serialises competing payment attempts. Resolving this class of bug is essential for wallets and applications that issue multiple payments in parallel without waiting for sequential confirmation.
sphere-sdk
AccountingModule, SwapModule, and CLI Overhaul
PRs #109, #110, and #111 add two major feature modules and a comprehensive CLI refresh to the Sphere SDK. #109 introduces AccountingModule, which manages the full invoice lifecycle including creation, status tracking, and payment attribution, enabling SDK consumers to build invoicing flows without custom state management. #110 delivers SwapModule, implementing a P2P token swap protocol backed by an escrow mechanism to ensure atomic exchange between counterparties. #111 overhauls the CLI with improved help output, shell completion support, and dedicated subcommands for the new accounting and swap capabilities, making all three features accessible from the command line. SDK versions 0.6.14 and 0.7.0 were cut in PRs #294 and #296 respectively to ship these changes into Sphere.

Sharded v2 Inclusion Proof Composition in Aggregator
PR #142 implements sharded inclusion proof composition for the v2 proof format in the Go aggregator. The change introduces sharding logic that partitions proof assembly across multiple shards, enabling the aggregator to scale proof composition horizontally rather than processing the entire tree in a single pass. This is a foundational scalability improvement for the network layer, as inclusion proof generation is on the critical path for transaction finality. By distributing the composition workload, the aggregator can sustain higher transaction throughput without proof latency becoming a bottleneck.
