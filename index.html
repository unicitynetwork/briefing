Astrid22 PRs merged
Kernel-enforced principal identity and capability model
PRs #932, #934, and #924 overhaul how principals are established and scoped at the kernel level. Unauthenticated socket connections now receive a no-capability anonymous stamp (#934), per-connection principals are kernel-verified on publish-as operations (#932), and new principals inherit nothing by default — opt-in inheritance via --inherit-from replaces the previous implicit propagation (#924). Together these changes close a class of privilege-escalation paths where an unverified caller could assume capabilities it was never granted. PR #930 extends the model further with permanent always-allow entries and a full admin token mint/revoke/list workflow.
astrid, sdk-rust
Sandbox hardening: credential and secret dir masking
PRs #937 and #932 ensure sensitive host directories never leak into sandboxed workloads. The operator's home credential directories are now masked inside the host_process sandbox (#937), and Astrid's own secret directories receive the same treatment as part of the kernel-principal work (#932). This prevents a compromised or malicious capsule from reading operator credentials via directory traversal within the sandbox boundary.
astrid
Hook handler API and file-injection spawn primitives
PRs #58 (sdk-rust) and #19 (capsule-hook-bridge) introduce a first-class hook handler surface: the astrid_sdk::hook module plus a #[hook] attribute macro, backed by a canonical hook-event-request shape published from the bridge. Read-only file-injection for sandboxed spawns lands in the WIT interface (#15, wit), the Astrid runtime (#890), and the Rust SDK (#57), allowing capsules to receive injected files at spawn time without write access. A documentation fix in #59 corrects the SDK reference to use the [subscribe] handler form rather than the superseded [[interceptor]] syntax.
astrid, sdk-rust, capsule-hook-bridge, wit
CLI enhancements: capsule install, clone, and subcommands
PR #926 makes manual capsule install default to all capsules while allowing --capsule to target a single one, and #922 fixes selection of the correct .capsule artifact when a monorepo ships multiple packages. Agent cloning arrives in #928 with agent create --clone <source> for full profile and state replication. Capsule-contributed CLI subcommands (astrid capsule <verb>) are wired up in #897, with capsule-cli PRs #27 and #28 binding each connection to a single principal with outbound demux and bridging run/result topics respectively. The Sage distribution (Claude on Astrid) gains its Distro.toml in #5 alongside a broker-correlation fix for MCP ingress responses in #4.
astrid, capsule-cli, sage

Sphere34 PRs merged
Address-switch identity bleed fully eradicated
A multi-PR campaign closes the address-switch bleed class that caused one user's wallet-api identity and inventory to contaminate another session. PR #585 introduces per-address wallet-api client isolation as the structural fix, #581 re-binds the shared wallet-api identity on re-visit switch, and a real-stack regression test (#588) validates the fix against the exact strand that surfaced the original incident. The sphere app is migrated to the new SDK in #377, and a dev.15 SDK pin with wake-socket liveness improvements ships in #374.
sphere-sdk, sphere
Payment reliability: replay, memos, and cross-session status
PR #590 delivers a batch of incident follow-ups covering replay backoff and poison-pill handling, stale-inventory surfacing, and correct in-flight balance accounting. Outgoing memos are fixed in #577 to read the canonical Sphere nametag rather than the empty minted-token store, a regression introduced at 6bd3058. Cross-session incoming payment-request status is now upserted correctly on re-surface and resolution events (#563), and receive() in #571 fresh-pulls deliveries instead of coalescing onto a stale background pump. Journey tests G2/G3/G5/G6 from the payments backlog are covered in #591.
sphere-sdk, wallet-api
Wake socket self-healing and realtime convergence
PR #567 adds a reconnect supervisor, liveness watchdog, and catch-up resync to the wallet-api wake socket so clients recover automatically from dropped connections. PR #565 ensures inventory and payment_request wake events are actually consumed, with an inventory poll backstop for missed pushes. A two-session same-owner realtime convergence test over the real wake socket lands in #569, and the M5.2 reconnect flake in wallet-api is resolved in #64 by gating on an offline-write barrier rather than a timing assumption.
sphere-sdk, wallet-api
Wallet-API observability, correctness, and CI hardening
Structured deposit logging, a new-recipient metric, and 4xx rejection logging arrive in wallet-api #60, with reject-reason recording added to /v1/mailbox/reject in #66. Terminal inventory-insert collisions are now returned as a typed 409 rather than a raw 500 (#61), and the flaky 57P01 pg-boss shutdown error is eliminated in #67. Auto-deploy of wallet-api staging on image publish with build-SHA verification is wired up across sphere-infra #9 and wallet-api #55, with the staging health check corrected in sphere-infra #13 to use the public /v1/health endpoint instead of the 403-protected /metrics route.

sif-docs
SHA-256 replaces DefaultHasher for API key storage
PR #68 swaps the API key hashing in semanticd from Rust's DefaultHasher to SHA-256. DefaultHasher is non-deterministic across process restarts and not collision-resistant, making it unsuitable for persistent credential storage. This change ensures API keys are stored and compared with a stable, cryptographically sound digest.
