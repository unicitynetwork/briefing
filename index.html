Oracle Pack Installer: Ship, Fix, and Harden
PRs #6–#14 across the oracles repo established and rapidly iterated on the Oracle pack installer pipeline, covering initial shipment (#6), preflight command validation (#10), checksum path acceptance (#9), pack ownership reconciliation (#12), and host-to-runtime workspace binding (#14). The public installer site (unicity-aos.github.io #8, #9) was updated to surface only live install methods and correctly detect coding-agent plugins. Homebrew tap PRs #7–#11 wired the tap into the signed stable AOS channel, fixed formula line length, bootstrapped the first audit pass, corrected the update branch target, and patched the offline initialization test. Collectively these PRs bring the Oracle distribution story from zero to a reliably installable, checksum-verified, Homebrew-auditable artifact.
oracles, unicity-aos.github.io, homebrew-tap
Oracle Release Train: v0.2.0 Through v0.2.5
Six consecutive release-cut PRs (#13, #15, #16, #18, #20 in oracles; #10, #11 in unicity-aos.github.io) marched Oracle through versions 0.2.1–0.2.5 in a single weekend, each paired with a published installer on the GitHub Pages site. CI hardening PRs #17 and #19 added pipeline gates that enforce immutable oracle releases and verify published release immutability after the fact, preventing accidental overwrites of shipped artifacts. PR #8 pinned Oracle v0.2.0 to Astrid v0.10.1, establishing the first explicit cross-component version lock. PR #21 fixed a Codex integration regression by correctly forwarding AOS runtime overrides through the oracle layer.
oracles, unicity-aos.github.io
AOS Community Edition: Nightly Train, Release Gates, and Meta Harness
PRs #30–#36 hardened the AOS CE release pipeline: a signed nightly train was introduced (#31), nightly tags were excluded from the stable push lane (#32), immutable-setting state parsing was fixed (#33), b3sum installation was moved before identity checks (#36), and the clean-home bootstrap gate result is now preserved across CI steps (#38). PR #34 pinned AOS 2026.1.0 to Astrid 0.10.0, mirroring the Oracle pinning strategy, while #35 added provenance testing for that pinned release. PR #41 established the AOS meta harness, a top-level test scaffold that coordinates validation across components, and PR #42 cut the 2026.1.2 release on top of it. PRs #37 and #39 fixed CLI shutdown confirmation and capsule identity usage in the release process respectively.
aos-ce
Capsule Skills and Oracle Host Discovery
PR oracles #15 introduced skill discovery across oracle hosts, enabling the runtime to enumerate capabilities exposed by remote oracle packs. aos-ce #44 complemented this by implementing discovery of installed capsule declarations within the CE environment, giving the skills subsystem visibility into locally installed capsules. These two PRs together wire up the full discovery path: local capsule declarations are surfaced by the CE, and the oracle layer aggregates skills from all connected hosts. The unicity-aos.github.io product site was rebuilt from scratch (PR #6) to reflect the new installer and distribution model that underpins this capability.
aos-ce, oracles

Concurrency Money-Safety: Serialize, Split, and De-Poison
PRs #688 and #689 address critical concurrency correctness in payment flows: #689 serializes the local-intents blob read-modify-write cycle identified as root cause A of a double-spend class of bugs, while #688 delivers three coordinated fixes covering split-strand races, payment double-pay, and knownSpends cache poisoning. PR #687 adds state-aware inventory reconciliation so tokens that are spent and later re-acquired are correctly re-integrated into the local state rather than silently dropped. wallet-api #108 exposes a per-row state hash in the inventory snapshot, giving callers a lightweight consistency signal without fetching full token data. Together these PRs significantly raise the bar for financial correctness under concurrent operations.
sphere-sdk, wallet-api
Send Result Contract, Mint Gate, and Proof-Poll Performance
PR #686 formally documents the send result contract, clarifying the semantics of the deliveryPending state and the mint gate that governs when a sent token is considered finalized. PR #685 improves payment throughput by introducing finer proof-poll cadence and parallelizing split mint operations, directly addressing latency issues tracked in #683 and #684. The unicity-claude-marketplace PR #12 rolls up sphere-connect 2.2.0, bundling the send result contract, mint gate semantics, and chainPubkey-first backend authentication into the marketplace integration. These changes tighten the observable behavior of the send flow and make it faster under load.
sphere-sdk, unicity-claude-marketplace
Connect Example: SDK Sync and Developer Documentation
sphere-sdk-connect-example #16 synchronizes the reference app to SDK 0.11.14 and surfaces the deliveryPending send contract in the UI, ensuring the example accurately reflects current SDK behavior. PR #18 adds a structured CLAUDE.md covering both the bot/ and backend-auth/ directories, closing issue #7 and giving AI-assisted contributors a clear map of the codebase conventions. These PRs keep the connect example in lockstep with the rapidly evolving SDK and lower the onboarding cost for developers integrating sphere-connect.
sphere-sdk-connect-example, sphere-sdk

SMT Algorithm Upgrade: Version 6a Inclusion Verification
state-transition-sdk-java #76 and state-transition-sdk-rust #14 both switch SMT inclusion verification from the older version 3o algorithm to version 6a, aligning the Java and Rust SDKs on a consistent proof format. Java PR #79 (merged as #80) additionally corrects the SMT bit ordering, fixing a subtle interoperability bug that would cause verification failures at the boundary between implementations. Java PR #77 removes the current version of nametag, cleaning up a construct that was coupled to the old SMT scheme. These changes are foundational: all higher-level proof verification in both ecosystems now runs on the same 6a algorithm, which is a prerequisite for cross-language proof portability.
state-transition-sdk-java, state-transition-sdk-rust
Mainnet Infrastructure: Terraform and Ansible Environment Bootstrapped
bft-terraform #20 creates a fresh mainnet environment in eu-west-1 by cloning the testnet2 configuration, introducing per-shard MongoDB replica sets appropriate for production data isolation. bft-ansible #26 ports the corresponding Ansible playbooks from testnet2, adding per-shard Mongo RS provisioning, genesis tooling, and monitoring stack configuration for the mainnet target. Together these two PRs represent the first concrete infrastructure definition for mainnet, giving the ops team a reproducible, IaC-managed environment that mirrors the battle-tested testnet2 topology while being sized and placed for production.
bft-terraform, bft-ansible
Aggregation Layer: RSMT Paper Revision 6
aggr-layer-paper PR #2, titled Rsmt6, updates the aggregation layer research paper with the sixth revision of the RSMT specification. This likely documents the algorithmic and structural changes underpinning the 6a SMT verification switch landing simultaneously in the Java and Rust SDKs. Keeping the paper synchronized with implementation changes ensures the formal specification remains the authoritative reference for implementors across all language targets.
aggr-layer-paper
