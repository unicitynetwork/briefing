Astrid6 PRs merged
Daemon reliability: SIGTERM, singleton races, orphan leaks fixed
PR #1174 addresses three intertwined daemon lifecycle bugs: SIGTERM starvation that left processes unkillable under load, a singleton startup race that allowed duplicate daemon instances, and orphan/restart leaks where child processes survived daemon restarts. Together these changes make the Astrid daemon safe to terminate and restart in production environments. This is a foundational stability fix for any workload that relies on clean daemon lifecycle management.
astrid
Copy-on-write capsule isolation for non-git workspaces
PR #1171 introduces real OS-level copy-on-write semantics for capsule snapshots in workspaces not managed by git, replacing the previous emulated or absent CoW behaviour. PR #1169 is the necessary complement, explicitly skipping the CoW overlay for git-managed workspaces where git's own object store already provides efficient snapshotting. Together these two PRs ensure each workspace type gets the optimal isolation strategy without double-layering overhead.
astrid
Capsule boot syncs contracts from daemon fleet
PR #1166 adds a boot-time refresh step in capsule-install that pulls canonical contracts directly from the daemon's fleet registry, ensuring newly installed capsules start with an up-to-date contract set rather than stale bundled definitions. This eliminates a class of startup errors caused by contract version skew between the capsule binary and live network state. The change is particularly valuable in long-running fleet deployments where contracts evolve between capsule installs.
astrid
Event routing capacity and dev-binary precedence fixes
PR #1160 raises the per-principal route queue cap from 256 to 16384, preventing queue saturation and dropped events for high-throughput principals in busy deployments. PR #34 in the sage plugin fixes binary resolution so that worktree or dev-local Astrid binaries are preferred over any PATH-installed version, eliminating accidental use of production binaries during development. Both are small but operationally significant correctness fixes.
astrid, sage

Sphere19 PRs merged
Wave 6 Phase-2 SDK v2 aggregator migration complete
PRs #646–#657 represent twelve successive wave-6 phase-2 commits that systematically migrate sphere-sdk from the legacy v1 aggregator API to the v2 surface. Key steps include: migrating the Profile pointer layer to the v2 aggregator API (#652), switching Profile persistence to UXF bundles (#653), purging remaining v1 backward-compat leaks (#654), deleting the legacy v1 import path entirely (#655), registering invoice tokens in payments after createInvoice (#657), and adding a process-wide trust-base fetch cache (#650) and soak-metrics harness (#649) for performance validation. The result is a clean v2-only SDK that is leaner, measurably faster, and ready for testnet2 soak testing.
sphere-sdk
Network-aware CLI and testnet2 nametag minting
PR #55 adds a --network flag to sphere-cli's on command (Phase-6 v2), and PR #52 threads that network value into Sphere.init so that v2 (testnet2) sessions can correctly mint nametags. PR #54 propagates the network parameter through the local host-manager for testnet2 tenant spawns, ensuring all spawned tenants inherit the correct network context. PR #53 removes the now-obsolete wallet migrate and legacy import commands as a direct companion to sphere-sdk#655's v1 path deletion, keeping the CLI surface clean and consistent with the v2-only SDK.
sphere-cli
Payments storm and storage toast reliability fixes
PR #646 fixes a request storm in heavy-wallet scenarios by introducing single-flight load() calls, incremental history hydration, and a client-side request timeout, resolving issue #642. PR #408 stops a separate storage:degraded toast notification storm in the Sphere frontend by bumping sphere-sdk to 0.11.5 and adding a toast cooldown guard. These two fixes together significantly reduce spurious network traffic and UI noise for users with large wallets or degraded storage conditions.
sphere-sdk, sphere

PR #410 fixes the onboarding flow to display the plan showcase consistently, matching the behaviour already present in the upgrade modal. The agentic-chatbot dependency bump in PR #16 ensures that downstream consumers of sphere-sdk also pick up the 0.11.5 fixes without manual intervention.
