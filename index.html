Fix principal selection in AOS status handling
PR #61 corrects the status endpoint in the AOS Community Edition to properly honour the selected principal rather than defaulting to an unintended one. This was a logic bug that could cause status responses to reflect the wrong principal's state, leading to misleading output for callers. The fix ensures the correct principal context is applied consistently when resolving status, improving correctness of the AOS identity model.
unicity-aos
Sphere9 PRs merged
Roll out sphere-sdk 0.12.0 across all consumers
PRs #19, #26, #28, #53, #14, #110, and #446 collectively propagate the @unicitylabs/sphere-sdk 0.12.0 release across every consumer repository in the Sphere ecosystem. The sphere-sdk-connect-example bump (PR #19) also includes a browser build fix affecting all four example projects. The sphere wallet (PR #446) uses a transitional 0.12.0-dev.1 pin and additionally gates the Connect feature to require sphere-sdk ≥ 0.12, enforcing a minimum capability baseline. This coordinated rollout ensures all surfaces are running a consistent SDK version, reducing integration drift and enabling features and fixes introduced in 0.12.0.
sphere-sdk-connect-example, sphere-backoffice, sphere-dev-portal, sphere-api, sphere-quest-frontend, wallet-api, sphere
Bump state-transition-sdk to 2.0.1
PR #693 in sphere-sdk and PR #110 in wallet-api advance the state-transition-sdk dependency to version 2.0.1. This patch release is pulled through into the sphere-sdk layer first and then consumed downstream by wallet-api, reflecting the correct propagation order through the dependency graph. Keeping the state-transition-sdk current ensures that state machine correctness fixes and any API surface changes in 2.0.1 are available to the wallet layer without version skew.
sphere-sdk, wallet-api
Surface auth and network errors in Quest frontend
PR #13 improves error visibility in the Quest frontend by surfacing actionable authentication and network errors that were previously swallowed or presented opaquely. The change also unifies the request path, consolidating how API calls are constructed and dispatched to reduce duplication and potential inconsistency. Together these improvements make failure modes debuggable by end users and developers alike, and lay groundwork for more reliable Quest API interactions going forward.
sphere-quest-frontend
