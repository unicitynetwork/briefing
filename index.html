Sphere5 PRs merged
Subscription Quota Gating and SGW Contract Alignment
PRs #404 and #402 complete the subscription enforcement layer for the wallet. PR #404 (Phase 4) introduces a dual-mode quota gate: a proactive pre-send block that prevents requests from leaving the client when quota is known to be exhausted, and reactive upgrade routing that intercepts 429 and 401 responses from the gateway to trigger plan-upgrade flows. PR #402 aligns the wallet-side subscription logic with the live SGW gateway contract, ensuring the client-side model matches what the gateway now enforces in production. Together these PRs close the loop between client-side quota awareness and server-side enforcement.
unicity-sphere
Grafana Dashboard and Alerting Single-Sourced from App Repo
PRs #28, #27, and #93 overhaul how the wallet-api Grafana dashboard and alert rules are managed. PR #28 restructures the sphere-infra pipeline so that the wallet-api dashboard and its alerts are single-sourced directly from the application repository, eliminating drift between infra and app definitions. PR #93 wires two new intent metrics into the dashboard and codifies their alert rules, giving operators visibility into intent-processing behaviour. PR #27 adds a companion documentation guide so engineers know the exact process for updating dashboard JSON and alert configurations going forward. The combined change reduces operational toil and makes dashboard-as-code the enforced pattern for this service.
unicity-sphere, unicity-sphere/sphere-infra, unicity-sphere/wallet-api

Token Split Phase 2 Across SDK and Yellow Paper
PRs #12 and #12 (respectively #12 in state-transition-sdk-rust and #8 in unicity-yellowpaper-tex) advance the second phase of the token-split work across both the Rust SDK and the protocol specification. The Rust SDK changes implement or extend the data structures and transition logic required to represent split tokens in the state machine. The yellow paper update keeps the formal specification in sync with the implementation, ensuring the written protocol definition reflects the new token-split semantics. Coordinating these two PRs together prevents specification drift and gives implementers a single consistent reference.
