Astrid15 PRs merged
Per-domain WIT ABI migration across kernel and SDKs
PRs #752, #44, and #6 collectively migrate the entire Astrid stack from a monolithic WIT interface to a per-domain WIT host ABI. The kernel (#752) establishes the new host-side contract, the Rust SDK (#44) follows suit and switches the target to wasm32-unknown-unknown, and the JS SDK (#6) completes the migration on the JavaScript side. This architectural shift cleanly separates capability domains at the interface level, improving modularity and enabling more precise capability grants. The breaking changes across all three repos land together, ensuring the full toolchain remains consistent.
astrid, sdk-rust, sdk-js
v0.7.0 release: publishability fixes and packaging
A cluster of release-blocking issues were resolved to enable the v0.7.0 workspace publish. PR #49 made astrid-sys cargo-publishable, PR #766 moved the bundled AppArmor profile inside the astrid-cli crate, PR #764 made astrid-capsule cargo-publishable, and PR #762 re-enabled the chrono clock feature on crates calling Utc::now(). With those blockers cleared, the release commits landed in #47 (sdk-rust) and #758 (astrid), and the Homebrew tap was bumped in #5. These fixes ensure end-users can install Astrid directly via cargo install astrid and via Homebrew.
astrid, sdk-rust, homebrew-tap
KV CAS interface: canonical mismatch signal fix
PR #10 in the wit repo introduces a breaking fix to the kv-cas WIT interface, establishing cas-mismatch as the single canonical signal for a compare-and-swap mismatch condition. Previously the mismatch error path was ambiguous, risking inconsistent handling across guest implementations. Standardising on one signal simplifies error handling logic in all language SDKs and makes the contract explicit in the interface definition itself. This is a foundational correctness fix ahead of broader KV-CAS adoption.
wit
JS SDK v0.1.0 launch: scope rename, CI, and release
The JavaScript SDK reached its initial public release through three sequential PRs. PR #10 renamed the npm scope from @astrid-os to @unicity-astrid, aligning the package identity with the project's current branding. PR #9 added a GitHub Actions release workflow that automatically generates release notes on tag push, establishing a repeatable publish process. PR #8 then committed the 0.1.0 release, making the JS SDK officially available to downstream consumers for the first time.

Sphere3 PRs merged
Profile monotonicity and detach consolidation correctness
PRs #262 and #270 address a pair of related correctness issues in the profile monotonicity subsystem. PR #262 fixes in-place merging of unknown bundles during the monotonicity check, preventing silent data loss when encountering bundle types not previously seen. PR #270 unblocks spec section C.2 by implementing detach consolidation and simultaneously correcting over-reporting in the monotonicity logic. Together these changes ensure the profile layer accurately tracks and reconciles state across a wider range of real-world bundle sequences. The fixes are tied to tracked issues #255 and #268 respectively.
sphere-sdk
HTTP-only IPFS mode for wallet and CLI clients
PR #267 introduces an HTTP-only IPFS mode targeted at wallet and CLI client environments where a full IPFS node is unavailable or undesirable. This mode allows the sphere profile layer to interact with IPFS purely over HTTP, removing the dependency on a local daemon for these lighter-weight client contexts. The change is tagged against issue #266 and expands deployment flexibility for end-user-facing tooling. It is a meaningful step toward making the sphere SDK usable in constrained or browser-adjacent environments without sacrificing IPFS-backed content addressing.
sphere-sdk
Unicity Network1 PRs merged
Revert: commit reward token ID in PoW block
PR #5 reverts a prior change and reinstates the requirement that the reward token ID be committed inside the proof-of-work block. This restoration likely reflects a protocol-level decision that the commitment is necessary for security or auditability of the reward mechanism. Encoding the token ID in the PoW block ensures it is cryptographically bound to the work performed, preventing post-hoc substitution. The change is captured in the yellow paper TeX source, meaning it updates the canonical protocol specification rather than an implementation.
