Capsule interface split into per-domain frozen packages
PR #7 breaks the monolithic astrid:capsule@0.1.0 WIT interface into discrete, domain-scoped frozen packages. The breaking change (feat!) signals a deliberate API boundary stabilisation, ensuring each domain can version independently without coupling to unrelated capability changes. Freezing packages at this granularity gives downstream consumers a stable contract per domain and reduces the blast radius of future interface evolution.

Sphere5 PRs merged
UXF token pipeline extended to unproven genesis and V5-pending state
PR #206 expands the UXF layer to accept tokens whose genesis has not yet been proven on-chain, and introduces visibility of V5-pending CAR bundles within that flow. This unblocks workflows where token issuance races ahead of proof availability, a common edge case in high-throughput payment streams. The change lays the necessary foundation for the V5 token resolution work that follows in PR #209.
sphere-sdk
Self-sufficient V5-pending bundles and resolveV5Token shape-reading
PR #209 builds directly on PR #206 to make payment bundles self-sufficient in the V5-pending state, embedding all data needed for resolution without requiring external lookups. The resolveV5Token function gains shape-reading capability, allowing it to introspect CAR structure and dispatch resolution logic based on the token's declared schema. Together these two PRs close issue #207 and complete the PR-B leg of the payments V5 feature track.
sphere-sdk
Hierarchical CAR path addressability across profile, bundle, and snapshot
PR #201 (closing issue #200) introduces a unified hierarchical addressing scheme so that profile, bundle, and snapshot CARs share a consistent, composable path structure. This eliminates ad-hoc path construction scattered across callers and makes content-addressed lookups predictable at every layer of the storage hierarchy. The change is a prerequisite for reliable snapshot diffing and any future cross-layer CAR traversal.
sphere-sdk
Conservative-mode finalize fix and recipient repair for proofless intermediates
PR #198 addresses issue #197 by correcting conservative-mode behaviour when an intermediate finalize operation carries no proof. Previously this path could leave a payment in an inconsistent state; the fix applies a recipient-repair step to restore integrity before the transaction is committed. The targeted scope of the fix avoids touching the optimistic path, keeping regression risk low while resolving a correctness issue in a critical payments edge case.
sphere-sdk
E2E scaffolding pattern rolled out to two additional test suites
PR #205 (closing issue #199) applies the scaffolding pattern established in PR #201 to two follow-up end-to-end test suites, standardising how fixtures, contexts, and assertions are structured across the suite. Consistent scaffolding reduces per-suite boilerplate and makes it easier to onboard new e2e scenarios without diverging conventions. This is a test-only change with no production code impact, but it materially improves maintainability as the e2e surface grows.
