Astrid1 PRs merged
Wildcard topic matching in capsule manifests
PR #702 extends the capsule manifest parser to accept trailing-* wildcards inside [[topic]] declarations. Previously, only exact topic strings were valid, forcing authors to enumerate every variant explicitly. The fix allows patterns such as 'events/*' to match entire topic subtrees, reducing manifest verbosity and the risk of missed declarations. This unblocks use-cases where topic namespaces are dynamically generated at runtime.

Unicity Network12 PRs merged
Relay throughput and contention fixes under load
PRs #17 and #14 in unicity-relay tackle kind-39002 write contention by increasing batch sizes, adding jittered exponential backoff, and introducing debounce logic to coalesce rapid updates, while PR #14 chunks event_tags batch inserts to stay beneath PostgreSQL's 65 535 parameter ceiling. On the tokens-relay side, PR #15 adds composite PostgreSQL indexes on (author, kind) columns to accelerate the query patterns that triggered the April 30 incident, and PR #16 closes the remaining observability gaps identified in that same incident. Together these changes reduce database lock contention, prevent parameter-overflow errors, and give operators the metrics needed to detect recurrence early.
unicity-relay, unicity-tokens-relay
WebSocket stability and connection health tuning
PR #12 bounds every WebSocket send with a 30-second WS_WRITE_TIMEOUT, preventing slow or stalled clients from holding goroutine resources indefinitely. PR #9 lowers the default ping_interval to 45 seconds and scales max_quiet_time proportionally, tightening dead-connection detection without causing spurious disconnects under normal latency. PR #10 fixes the db_writer rate limiter to use tokio::time::sleep instead of a busy-wait or incompatible timer, ensuring the limiter behaves correctly inside the async Tokio runtime. These three changes collectively improve relay stability under high client-count scenarios.
unicity-tokens-relay
Relay observability and broadcast metrics
PR #11 exposes a new nostr_broadcast_lagged_total Prometheus counter that increments whenever a broadcast channel receiver falls behind and events are dropped. This metric was previously invisible, making it impossible to distinguish slow-consumer drops from normal delivery in dashboards. Combined with the observability additions in PR #16, operators now have end-to-end visibility from ingest through broadcast to the database writer. Having discrete counters per failure mode shortens mean-time-to-diagnose during future incidents.
unicity-tokens-relay
Node listener and routing hardening
PR #11 in unicity-node raises the TCP listen backlog from a hardcoded 20 to SOMAXCONN, allowing the kernel to queue far more pending connections before refusing them under burst traffic. PR #10 bundles additional node fixes from the 'Addnode' work stream, addressing connection-management edge cases identified during recent load testing. In aggregator-subscription, PR #30 adds routing based on the X-State-ID HTTP header, enabling the service to skip full body parsing when the state identity is already present in the header, reducing CPU overhead on the hot path. PR #15 in unicity-relay adds a CI workflow that runs Go tests on every push and restricts Docker image builds to the master branch, tightening the feedback loop while avoiding unnecessary registry churn.
unicity-node, aggregator-subscription
