Astrid3 PRs merged
Per-Principal Identity Profiles via profile.toml
PR #665 introduces PrincipalProfile, a per-principal identity construct backed by a profile.toml configuration file, closing issue #663. This gives each principal a structured, declarative profile that can be resolved and loaded at runtime, laying the groundwork for fine-grained principal-level customization. The addition of profile.toml as the canonical descriptor means principal metadata is now version-controllable and portable across deployments. This is a foundational capability for multi-tenant and multi-principal Astrid environments.
unicity-astrid
Per-Invocation Secret Store and Scoped Capsule Logging
PR #662 introduces a per-invocation SecretStore, ensuring that sensitive material is scoped strictly to the lifetime of a single capsule invocation rather than shared across calls. Alongside this, capsule_log is re-scoped so that log output is isolated per invocation, preventing cross-invocation log bleed and improving auditability. Together these changes harden the capsule execution model by enforcing strict isolation boundaries around both secrets and observability data. This is critical for security posture in multi-tenant workloads where invocation-level confidentiality must be guaranteed.
unicity-astrid
Per-Invocation home:// and /tmp/ VFS Resolution
PR #659 implements per-invocation resolution of the home:// and /tmp/ virtual filesystem paths within the capsule VFS layer. Previously these paths could resolve to shared or persistent locations, creating the risk of state leakage between capsule invocations. With this change, each invocation receives its own isolated home and temporary directory namespaces, fully consistent with the per-invocation isolation model being built across PRs #662 and #665. This ensures that filesystem side-effects from one invocation cannot be observed or tampered with by another, strengthening the overall capsule sandbox guarantees.

Unicity Network3 PRs merged
Timelock Predicate Encoding Formalized in Yellowpaper
PR #3 in the yellowpaper repository formally specifies the encoding of timelock predicates, adding rigorous mathematical and structural definitions to the Unicity protocol documentation. Timelock predicates are a key primitive for time-bound transaction logic, and their precise encoding is essential for interoperability and correct implementation across clients. Formalizing this in the yellowpaper ensures that all implementors share a single normative reference, reducing the risk of divergent interpretations. This contribution advances the completeness of the Unicity protocol specification.
unicity-yellowpaper-tex
ShardID Verification Added to Unicity Certificates
PR #5 in bft-go-base adds shardID verification to Unicity Certificates (UCs), ensuring that a certificate is explicitly bound to the shard it was issued for. Without this check, a certificate from one shard could potentially be replayed or misapplied in the context of another shard, representing a cross-shard integrity risk. The addition of this verification step strengthens the UC validation pipeline and aligns certificate semantics with the sharded architecture of the Unicity network. This is an important correctness and security improvement for the BFT base layer.
bft-go-base
CBOR Tag Support in Aggregator Serialization
PR #143 introduces CBOR tag support to the aggregator-go service, updating the serialization layer to correctly encode and decode typed CBOR structures using explicit tags. CBOR tags enable self-describing, strongly-typed binary data, which improves interoperability with other components and external systems that consume aggregator output. This change likely resolves ambiguity in previously untagged CBOR payloads where type information had to be inferred from context. Proper CBOR tagging is a step toward a more robust and standards-compliant data interchange format across the Unicity network stack.
aggregator-go
