Full SDK 0.6.0 migration across all 17 capsules
PRs #7, #13, #9, #10, #10, #11, #15, #10, #15, #3, #14, #17, #10, #12, #10, #9, and #5 collectively migrate every capsule in the unicity-astrid organisation to SDK 0.6.0, driven entirely by @joshuajbouw. The upgrade adopts the wasmtime Component Model, replacing the previous module-based execution boundary with a composable component architecture that enables fine-grained interface linking at the Wasm layer. WIT-driven types replace ad-hoc serialisation contracts, meaning each capsule now expresses its public API through machine-verifiable WebAssembly Interface Type definitions. This eliminates an entire class of runtime type-mismatch errors and makes cross-capsule composition explicit and toolchain-enforced. The breadth of the change — spanning agents, identity, memory, routing, session, shell, UI, storage, and both OpenAI integration surfaces — establishes SDK 0.6.0 as the new baseline for all future capsule development.

Fix broken SwapModal UI component
PR #292 by @igmahl resolves a defect in the SwapModal component within the sphere frontend. The fix addresses incorrect rendering or interaction behaviour that was impacting the token-swap user flow inside the Unicity Sphere interface. Isolating the fix to the modal component ensures the surrounding swap infrastructure remains unaffected while restoring correct UI state management. Though scoped to a single component, SwapModal sits on a critical user path, making this a meaningful stability improvement for end users.

Add payment functionality to Java state-transition SDK
PR #54 by @martti007 implements the payment feature tracked in issue #51, adding payment transaction support to the state-transition-sdk-java library. This introduces the data structures, state-transition logic, and likely signing or validation plumbing required to represent and process payment operations within the SDK's transaction model. Shipping payment capability at the SDK level means downstream Java integrators can construct and submit payment state transitions without building bespoke transaction handling themselves. The change extends the SDK's coverage of core Unicity network primitives and moves the Java integration surface closer to parity with other language SDKs.
