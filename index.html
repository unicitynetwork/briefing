Sphere13 PRs merged
Self-healing payment coin selection and mailbox resilience
PRs #622, #624, and #626 deliver a multi-part overhaul of the payments subsystem in sphere-sdk. #626 (P3 of issue #625) introduces self-healing coin selection that automatically demotes already-spent sources and re-plans the transaction, preventing stuck payment flows caused by stale UTXO state. #624 adds batched claim/reject processing for incoming mailbox pages (issue #623), reducing round-trips when clearing large mailbox backlogs. #622 (issue #621, §3.1 compliance) ensures that a failure in the recipient-side mailbox state machine never propagates as a sender-visible error, correctly isolating fault domains between sender and receiver.
sphere-sdk
Configurable rate limits and quota disabling in wallet-api
PRs #80 and #82 in wallet-api introduce the ability to disable rate limits and entry-count caps by setting thresholds to zero or below. #82 makes the auth, deposit, and write rate-limit families honour a ≤0 sentinel to disable them entirely (issue #81), enabling controlled high-throughput scenarios. #80 extends the same zero-disables-cap semantics to per-recipient, per-sender, and per-owner quota counters, making all quota axes uniformly configurable. sphere-infra PRs #22 and #23 apply these new capabilities to the deployed wallet-api, raising per-owner rate limits by 10× and explicitly disabling the §5.5 entry-count caps in the testnet configuration.
wallet-api, sphere-infra
MongoDB reliability and /ready endpoint for sphere-quest
PR #37 in sphere-api adds fail-fast driver timeouts to the MongoDB client and introduces a /ready HTTP endpoint that returns an error when no primary is detected, enabling fast failure signalling to orchestrators. sphere-infra PR #18 complements this with a mongod liveness watchdog that automatically recovers a wedged mongod process without manual intervention. These two changes together form an auto-recovery loop: the watchdog restarts a stuck node while the /ready endpoint lets upstream load balancers and alerting systems detect and react to primary loss immediately.
sphere-api, sphere-infra
Grafana Alloy observability stack for sphere-quest
PRs #17, #18 (shared with mongo theme above), #19, #20, and #21 build out a full observability layer for the sphere-quest service. #17 deploys Grafana Alloy to collect host and container metrics, with a unified dashboard covering both staging and production. #19 adds a Blackbox Exporter uptime probe against the quest /ready endpoint with an alert wired to a working contact point. #20 fixes the probe by adding environment labels and corrects the alert routing that was previously targeting a broken contact point. #21 patches the sync.sh reload script to also restart the Alloy process, ensuring configuration changes take effect without manual intervention.

Testnet2 ECS gateway deployment and blue/green service
PRs #10, #11, #9 in bft-terraform and #18, #19 in bft-ansible establish a full ECS Fargate gateway deployment for testnet2. #10 creates the ECS Fargate service configured for Track 2 blue/green deployment without cutting over live traffic yet. #11 adds the canary ALB listener rule that associates the ECS IP target group, enabling controlled traffic shifting. #9 switches the ALB health check target from /health to /metrics, aligning with the actual endpoint exposed by the gateway image. bft-terraform, bft-ansible
Testnet2 gateway autoscaling observability via CloudWatch
PRs #12 and #20 add end-to-end observability for ECS gateway autoscaling on testnet2. #12 in bft-terraform enables Container Insights on the ECS cluster and grants the necessary CloudWatch read permissions so scaling metrics are available to monitoring tooling. #20 in bft-ansible deploys a Grafana Alloy CloudWatch exporter that scrapes ECS scaling metrics and forwards them to the existing metrics pipeline, giving operators a live view of gateway capacity changes. Together these changes close the observability gap that existed once the gateway moved from static EC2 instances to dynamically-scaled ECS tasks.
bft-terraform, bft-ansible
Aggregator autoscale readiness, OOM hardening, multi-arch images
PRs #48 and #49 in aggregator-subscription prepare the aggregator service for production-grade horizontal scaling. #48 (issue #47) is a comprehensive hardening PR that introduces shared rate limiting across instances, native-memory and OOM safeguards, graceful drain on shutdown, and stateless admin sessions so any instance can serve administrative requests without affinity. #49 extends the CI pipeline to publish multi-architecture Docker images, adding linux/arm64 alongside the existing amd64 build to support deployment on AWS Graviton instances, reducing compute cost at scale.
aggregator-subscription
CMS database password self-management to prevent rotation outages
PR #9 in website-next addresses recurring outages caused by automatic rotation of the CMS database master password by an external system. The change moves to self-managed password handling, giving the application explicit control over when and how the credential is rotated. This eliminates the race condition where the secrets manager rotated the credential mid-session without the application being aware, causing connection failures. The fix improves availability of the public-facing website CMS without requiring changes to the underlying database infrastructure.
