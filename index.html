Sphere5 PRs merged
Subscription quota gating and SGW contract alignment
PRs #404 and #402 together close out the subscription enforcement layer. PR #402 aligns the wallet client with the live SGW gateway contract, ensuring request shapes and auth handshakes match what the production gateway now expects. PR #404 (Phase 4) layers on a dual-mode quota gate: a proactive pre-send block that prevents messages from being dispatched when the quota is already known to be exhausted, and reactive upgrade routing that intercepts 429 and 401 responses and escalates them through the correct subscription-upgrade flow. Together these changes make quota enforcement a first-class, production-safe capability rather than a best-effort client-side hint.
unicity-sphere
Wallet-API Grafana dashboard and alerts single-sourced
PRs #28, #27, and #93 establish a single-source-of-truth approach for the wallet-api observability stack. PR #93 wires two new intent metrics into the Grafana dashboard and defines corresponding alert rules inside the wallet-api app repository. PR #28 (sphere-infra) plumbs those definitions so the wallet-api dashboard and its alerts are generated from the app repo rather than maintained separately in infra, eliminating drift. PR #27 accompanies this with an authored guide explaining how engineers should update the dashboard and alert rules going forward, lowering the operational burden for future metric additions.
unicity-sphere, sphere-infra, wallet-api

Token split phase 2 across Rust SDK and yellowpaper
PRs #12 and #12 (respectively #12 in state-transition-sdk-rust and #8 in unicity-yellowpaper-tex) both carry the 'Token split 2' milestone, indicating a coordinated second phase of work partitioning token representations or semantics. The Rust SDK PR advances the state-transition logic to handle split token structures at the code level, while the yellowpaper PR updates the formal specification document to reflect the same structural changes, keeping implementation and specification in lockstep. Keeping these changes paired ensures that the canonical protocol description and the reference implementation do not diverge as the token model evolves.
