Astrid52 PRs merged
Tool dispatch reborn as IPC interceptor pipeline
PRs #591 and #593 ripped out the legacy ToolDef and CronDef abstractions along with the kernel's dedicated tool dispatch path, replacing them with a pure IPC convention. The Rust SDK followed in #17, rewiring the #[astrid::tool] macro to emit interceptor arms instead of tool registration stubs. Interceptors now carry explicit priority ordering (#581) and support veto semantics (#587), letting any capsule in the chain short-circuit execution before it reaches the target. This shift unifies all cross-capsule communication under a single, composable interception model rather than a parallel tool-specific subsystem.
astrid, sdk-rust
Capsule install, remove, and dependency graph CLI
A full capsule lifecycle CLI landed across PRs #556, #559, #569, #571, #577, #579, and #586. Capsules can now be fetched directly from GitHub release WASM assets, installed with distro-driven configuration parsed from Distro.toml (#567), and removed with dependency safety checks that refuse to break dependent capsules. The artifact store (bin/ and wit/) is append-only (#579) and uses content-addressed WIT storage (#575) so installs and removals never clobber shared interface files. A tree visualizer (#571) exposes the full dependency graph, and conflict detection (#586) catches export collisions at install time before they reach the runtime.
astrid
WIT interface contracts and kernel import/export validation
PR #561 replaced the informal provides/requires fields in capsule manifests with structured [imports] and [exports] tables, giving the kernel a machine-readable contract for every capsule boundary. All ten first-party capsules followed with matching declarations in their own PRs (#5–#8 across repos), and the Rust SDK gained bundled WIT definitions for all standard contracts (#16). The kernel now validates these imports and exports at boot (#563), rejecting capsules whose declared interfaces don't match what the host ABI actually exposes. The host ABI itself was fully documented in a 51-function specification rewrite (#573), making the contract authoritative rather than implicit.
astrid, sdk-rust, capsule-identity, capsule-registry, capsule-hook-bridge, capsule-context-engine, capsule-prompt-builder, capsule-router, capsule-session, capsule-react, capsule-openai-compat
Principal resolution, audit chains, and capsule pipelines
Per-invocation principal resolution (#545) and follow-up plumbing fixes (#547, #553) ensure every request carries an authenticated identity through the kernel before dispatch, with a cap on known_principals growth and automatic retry on provision failure. Audit chains were split per-principal (#554) so each identity accumulates its own tamper-evident log rather than sharing a single global chain. The global:// capability scheme was renamed to home:// in capsule-fs (#5) and capsule-skills (#6) for clearer semantic intent, and capsule-identity renamed its topics from identity.v1 to spark.v1 (#8). Fourteen capsules received WASM release pipelines via CI PRs, and the new astralis meta-repo was bootstrapped with an identity capsule version fix (#2) and license attribution (#1).
astrid, capsule-fs, capsule-skills, astralis, capsule-identity

Nostr relay correctness: member count and stale detection
PR #8 in unicity-relay fixed member_count to be sourced from kind 39000 GROUP_METADATA events rather than a potentially stale or incorrect location, aligning the relay with the NIP spec for group metadata. PR #6 in nostr-js-sdk addressed false stale detection in the relay client, preventing the SDK from incorrectly marking live relay connections as inactive and dropping them prematurely. Together these fixes improve the reliability of group-aware Nostr relay communication across the Unicity network layer.
unicity-relay, nostr-js-sdk
Aggregator: continuous precollector and empty-round lag fix
PR #136 by @jait91 extracted the continuous child precollector into its own isolated component within the aggregator-go service, improving separation of concerns and making the collection loop easier to reason about independently. The PR simultaneously fixed a lag condition that occurred during empty rounds, where the absence of new data caused the aggregator to stall or delay unnecessarily. This improves throughput consistency for the aggregation pipeline even during low-traffic periods.
