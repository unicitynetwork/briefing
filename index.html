CORS, Auth & Gateway Hardening
PR #34 introduces an environment-driven CORS allowlist with cors_enabled as a master switch, replacing any hardcoded origins and making the security posture configurable per deployment. PR #28 lands the full SIF Gateway compose stack with real authentication and integrated observability, turning what was previously a skeleton into a production-ready service. Together these PRs close the two most critical surface-area risks on the SIF ingress path and establish a repeatable deployment pattern for the gateway.
sif
Relay Performance: Caching, Denormalization & Metrics
PR #26 warms membership and role caches from kind-39001 and kind-39002 snapshots at startup, eliminating cold-cache latency spikes on group permission checks. PR #24 denormalizes the event kind directly onto the event_tags table so the tag CTE can pre-filter by kind before joining, significantly reducing query fan-out on high-volume filter patterns. PR #22 splits database time from drain time in the metrics pipeline and widens histogram buckets to 30 seconds, giving operators a cleaner separation between storage latency and queue backpressure.
unicity-relay
Observability & Real-Time Broadcast Visibility
PR #31 replaces placeholder dashboard values in SIF with real P50, P98, and Max latency figures sourced directly from the audit log, making SLA monitoring actionable for the first time. PR #24 in unicity-tokens-relay adds delivery-side observability instrumentation across the live broadcast path, exposing per-message timing and drop metrics that were previously invisible. These two PRs establish end-to-end latency visibility from gateway ingress through token relay delivery.
sif, unicity-tokens-relay
Security Detection, SDK Foundations & Infrastructure Fixes
PR #2 completes YARA detection wiring in SIF, enabling pattern-based threat scanning on inbound payloads as a first-class pipeline stage. PR #1 in unicity-faucet deduplicates concurrent nametag resolution requests, preventing thundering-herd conditions that were tripping the relay subscription cap under load. PR #53 in state-transition-sdk-java lays down the general architectural structure for the Java SDK and implements token minting and transfer functionality, unblocking downstream Java integrations. PR #3 in website-next ships a batch of website content updates under the Feature/website-content branch.

✔ What's moving

94% completion reached across 51 items
48 of 51 items are done, demonstrating sustained delivery momentum across the project. The team has closed the vast majority of TGE requirements ahead of the April 30 deadline.
JS SDK token validation work in progress
@martti007 is addressing the TokenId length validation gap in state-transition-sdk-js. This fix is essential for release integrity — accepting non-32-byte token IDs in production would be a correctness defect.
Unicity data structure versioning underway
@ahtotruu is adding version tags to Unicity data structures in unicity-ids. Versioning is a hard dependency for forward compatibility and safe TGE deployment.
BFT sharding task assigned to core team
The BFT-based sharding implementation in .github is assigned to @ahtotruu, lploom, and martti007, indicating team awareness and nominal ownership of the highest-complexity remaining item.

⚠ What worries

BFT sharding scope is release-critical and incomplete
Implementing BFT-based sharding in Unicity aggregation (.github) is the most architecturally significant open item and has not been marked done with 4 days to deadline. This is not a minor fix — BFT consensus sharding is a foundational change that cannot be safely rushed or partially shipped.
All 3 open items owned by same 2-3 people
@ahtotruu owns two of the three open tasks (unicity-ids versioning and BFT sharding), and @martti007 owns or co-owns two tasks. Any personal blocker, review bottleneck, or merge conflict affecting either individual stalls multiple critical path items simultaneously.
Zero days of buffer before April 30 deadline
The deadline is today. All three remaining items must be completed, reviewed, and merged with no remaining calendar slack. Any rework, failed CI, or review cycle pushes at least one item past the release date.
BFT sharding may block JS SDK or versioning merge
If the BFT sharding work in .github touches aggregation interfaces or data structure contracts, it creates an undeclared dependency on the unicity-ids versioning task (@ahtotruu). Merging versioning tags before sharding stabilizes the schema risks merge conflicts or API breakage in the JS SDK.
JS SDK validation fix has no stated reviewer
The state-transition-sdk-js TokenId validation fix (@martti007) has no explicit reviewer listed. With @martti007 also co-assigned to BFT sharding, self-review pressure is high and the fix may ship without independent validation of edge cases.
BFT sharding is not a TGE-scoped deliverable by nature
BFT-based sharding in a consensus aggregation layer is typically a multi-sprint architectural initiative. Its presence as a TGE open item with no done status at deadline suggests either scope creep, underestimation, or a risk that it will be shipped incomplete or stubbed out — all of which carry post-TGE correctness risk.
