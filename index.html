Capsule lifecycle: safe removal with purge control
PR #646 introduces a critical fix to preserve environment configuration when removing a capsule, preventing accidental data loss during teardown. A new --purge flag gives operators explicit control over whether env config is wiped, making destructive operations intentional rather than implicit. This improves operational safety for teams managing multiple capsule deployments.

Telegram capsule: reliability and resilience hardening
PRs #12 and #14 address API-layer robustness by adding exponential backoff on Telegram API failures and enforcing HTTP status code validation alongside IPC dropped-message monitoring. PR #17 introduces TTL-based cleanup for turns and pending_approvals, closing a memory leak that would degrade long-running capsule instances. Together these changes make the Telegram capsule production-worthy under real-world load and failure conditions.

Telegram capsule: correctness fixes across core logic
PR #11 fixes multi-user approval and elicitation routing, ensuring messages are delivered to the correct participant in multi-party conversation flows. PR #18 resolves enum parsing bugs, trims Capsule.toml bloat, fixes link double-escaping, and removes dead code, tightening the overall codebase.
