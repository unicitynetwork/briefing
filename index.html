Astrid10 PRs merged
Canonical WIT repository becomes single source of truth
PRs #3 and #4 in the wit repo establish the canonical home for all WebAssembly Interface Type definitions, introducing host/astrid-capsule.wit as the authoritative kernel-to-capsule ABI and adding a principal field to the ipc-message record. SDK consumers are then migrated to consume from this canonical source: sdk-js #1 and sdk-rust #38 and #39 all switch their contract WIT imports away from local copies to unicity-astrid/wit. This consolidation eliminates drift between language-specific SDK definitions and the core ABI, making future interface changes a single-point update.
wit, sdk-js, sdk-rust
Principal propagation plumbed through IPC uplink
sdk-rust #37 introduces the ipc-publish-as WIT binding enabling callers to stamp a message with an explicit uplink principal, while capsule-cli #12 wires this into the CLI so client-asserted principals are forwarded via publish_json_as (feature gap F-G). capsule-cli #13 extends this further by emitting principal-stamped connect and disconnect lifecycle events, giving downstream consumers a verified identity trail across session boundaries. Together these PRs close the principal-propagation gap identified in the F-G feature track and establish end-to-end principal visibility from client connection through IPC routing.
wit, sdk-rust, capsule-cli
Sandbox hardening: AppArmor profile and bypass prevention
astrid #742 is a breaking fix that closes issue #655 by refusing silent sandbox bypasses that previously allowed capsules to operate outside confinement without explicit failure. The PR ships a production AppArmor profile alongside an astrid setup utility, giving operators a concrete mandatory-access-control baseline. capsule-cli #11 complements this by fixing per-session response routing and restoring the admin IPC allowlist, addressing feature-track items F-H and Gap 2. Collectively these changes move the sandbox from a best-effort boundary to an enforced one with proper tooling for deployment.
astrid, capsule-cli

Deterministic rule-file loading in SIF engine
PR #45 changes the SIF rule loader to process rule files in a deterministic, sorted order rather than relying on filesystem enumeration order, which varies across operating systems and container runtimes. This eliminates a class of environment-dependent bugs where rule evaluation order, and therefore policy outcomes, could differ between development and production. Deterministic ordering is a prerequisite for reproducible rule-set auditing and consistent test outcomes.
sif
Network separation documented in yellow paper
PR #4 adds a network separation section to the Unicity yellow paper LaTeX source, formally specifying how distinct network domains or partitions are delineated within the protocol. Formalising this in the yellow paper anchors subsequent engineering decisions to a reviewed specification and provides an auditable reference for external stakeholders. This is a documentation-layer change with direct implications for protocol design conformance.
