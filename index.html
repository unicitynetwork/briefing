Capsule lifecycle and env config hardening
PR #646 fixes a long-standing issue where environment configuration was silently discarded when removing a capsule. A new --purge flag gives operators explicit control over whether env config is wiped on capsule removal, separating destructive from non-destructive teardown. This prevents accidental config loss during routine capsule lifecycle operations and aligns removal semantics with user intent.
unicity-astrid
Telegram capsule reliability and correctness fixes
PRs #11 through #19 collectively harden the capsule-telegram integration across several dimensions. PR #11 fixes multi-user approval and elicitation routing so messages reach the correct user in concurrent sessions. PR #12 introduces exponential backoff on Telegram API failures, preventing thundering-herd retries under API degradation. PRs #14 and #15 add HTTP status code validation, IPC dropped-message monitoring, and proper propagation of uplink::register errors so failures surface rather than silently pass.
capsule-telegram
Telegram capsule memory and parsing cleanup
PR #17 introduces TTL-based cleanup for turn state and pending_approvals, closing a memory leak that would grow unbounded in long-running sessions. PR #18 addresses enum parsing correctness, trims unnecessary fields from Capsule.toml to reduce configuration bloat, fixes double-escaping in link formatting, and removes dead code. PR #19 resolves all remaining findings from the second code review, indicating the capsule-telegram module is approaching production readiness.
