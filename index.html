Capsule authoring scaffold and Sage distro integration
PRs #942 and #9 together deliver end-to-end capsule authoring support: #942 adds the `astrid capsule new` CLI scaffold complete with toolchain preflight checks and an AUTHORING.md guide, while #9 registers a dedicated capsule-authoring forge inside the Sage distro. These changes give developers a first-class, guided path for creating and publishing capsules without manual setup. The toolchain preflight ensures that required dependencies are validated before any scaffolding occurs, reducing onboarding friction.
astrid, sage
Daemon stability: PID-safe orphan kill and shim reconnect
PR #941 addresses two related daemon lifecycle bugs: orphan processes left behind during stop/restart cycles are now killed using a PID-reuse-safe strategy, preventing accidental termination of unrelated processes that inherit a recycled PID. The same PR also adds shim reconnect logic that triggers automatically when a response loss is detected, improving resilience against transient IPC failures. Together these fixes harden the daemon against the class of subtle race conditions that emerge during rapid restart sequences.
astrid
Keyless principal keypair backfill and home introspection furniture
PR #946 fixes a regression where existing principals created without a keypair were left in an inconsistent state when a new agent was created on their behalf; the fix backfills the missing keypair at agent-create time. PR #940 ensures that per-principal home directories are fully materialized with the expected introspection furniture during capsule install, so that runtime self-inspection APIs have a consistent layout to query. Both fixes address gaps in the principal lifecycle that could cause subtle failures in deployed environments.
astrid
Sage MCP broker observability and claude plugin scoping
PR #8 introduces structured request-lifecycle observability into the Sage MCP broker, emitting discrete lifecycle events for each request so that operators can trace the full path of a message through the broker without relying on unstructured logs. PR #7 scopes the claude-code plugin into a dedicated custom `claude` group, isolating its configuration and permissions from the default plugin namespace. These two changes collectively improve operational visibility and access-control hygiene within the Sage plugin layer.
sage

Sphere Connect compatibility gate and canonical network standard
PR #598 is the core delivery, implementing a protocol-and-network compatibility gate in Sphere Connect along with a canonical network standard definition that all connect implementations must satisfy. PR #4 in the marketplace repo documents the resulting v2 compatibility gate and the network standard, giving downstream integrators a clear reference for what is required to pass the gate. PRs #599 and #600 are cleanup follow-ons that remove the SDK_VERSION drift guard test which had been blocking npm publish since the new gate supersedes the old version-skew check. Together these changes give the Sphere SDK a stable, versioned interoperability contract.
sphere-sdk, unicity-claude-marketplace
NIP-17 self-wrap opt-out for transport layer (M8 mitigation)
PR #558 implements a NIP-17 self-wrap opt-out flag in the Sphere SDK transport layer, tracked as mitigation item M8. NIP-17 self-wrapping adds an extra layer of gift-wrap encryption around messages sent to oneself, which is correct for privacy but introduces overhead and complexity in specific relay or SDK-internal flows where the sender and recipient are the same key. The opt-out allows transport consumers to bypass self-wrapping in those controlled contexts while leaving the default behavior intact for all external messaging. This is a targeted mitigation rather than a protocol change, preserving NIP-17 compliance for the general case.
sphere-sdk

SMT corruption prevention during HA leadership churn
PR #158 in aggregator-go fixes a critical correctness issue where Sparse Merkle Tree state could become corrupted when high-availability leadership changed hands rapidly. During leadership churn, concurrent or out-of-order writes from an outgoing leader could interleave with writes from the incoming leader, producing an inconsistent SMT that silently undermines proof validity. The fix introduces guards that fence SMT mutations to the current leader epoch, ensuring that any in-flight operations from a deposed leader are discarded before the new leader begins writing. This protects the integrity of the aggregator's cryptographic accumulator under realistic HA failure scenarios.
aggregator-go
Clippy default settings enforcement in semanticd
PR #69 resets the clippy lint configuration in semanticd to Rust's default settings, removing any project-specific overrides that had been suppressing or altering standard lints. Aligning with default clippy settings increases consistency with the broader Rust ecosystem and ensures that common code-quality warnings are not silently hidden. This is a code-hygiene change with no functional impact, but it lays groundwork for enforcing a clean lint baseline across the unicitynetwork Rust codebase.
semanticd
