Capsule lifecycle and environment config safety
PR #646 fixes a regression where environment configuration was lost when removing a capsule, and introduces a --purge flag to give operators explicit control over full teardown versus soft removal. This distinction matters for production deployments where env config must survive capsule churn. The change hardens the capsule lifecycle contract and reduces the risk of silent misconfiguration after remove operations.
unicity-astrid
Telegram capsule reliability and correctness overhaul
PRs #11 through #19 represent a sustained hardening pass across the capsule-telegram integration. Core correctness fixes include multi-user approval and elicitation routing (#11), exponential backoff on Telegram API failures (#12), HTTP status code checking and IPC dropped-message monitoring (#14), and error propagation for uplink::register failures (#15). Memory safety was addressed in #17 via TTL-based cleanup for turns and pending_approvals, eliminating a long-lived leak. Code quality issues resolved in #18 cover enum parsing correctness, Capsule.toml bloat reduction, link double-escaping, and dead code removal. PR #19 closes all remaining findings from the second code review, bringing the capsule to a reviewable and shippable state.
