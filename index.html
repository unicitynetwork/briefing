Astrid7 PRs merged
Dynamic ingress consent replaces static allow-lists
PRs #918 and sage#2 replace hardcoded allow-lists and trusted_ingress_ids with runtime elicit-on-block ingress consent in both the CLI-MCP and sage-MCP layers. Instead of pre-approved identities, the broker now prompts for consent when an ingress attempt is blocked, giving operators fine-grained dynamic control. This change touches both the Astrid CLI shim and the companion Sage service, ensuring the two MCP surfaces behave consistently. The shift from static configuration to interactive consent is a foundational trust-model change for the broker ingress path.
unicity-astrid, sage
Deterministic capsule identity via UUID v5
PRs #911 and #914 together harden capsule identity generation by switching to deterministic UUID v5 derived from a name, and by introducing a dedicated namespace for those derivations. Previously, identity could be non-deterministic across restarts, risking inconsistent capsule resolution. The dedicated namespace (PR #914) prevents UUID collisions between capsule identities and other v5-namespaced objects in the system. These two fixes together make capsule identity stable and reproducible across daemon lifecycles.
unicity-astrid
MCP shim reconnection and kernel-router isolation
PR #916 fixes the MCP serve shim so it automatically reconnects after a daemon restart, eliminating the need for manual intervention when the underlying daemon cycles. PR #909 addresses a correctness issue in the kernel-router by filtering out non-kernel requests that arrived on the shared request namespace, preventing misrouted messages from polluting kernel handling. Together these two fixes improve the operational resilience of the MCP layer and the routing correctness of the kernel path.
unicity-astrid
WASM-safe backoff in capsule-cli proxy accept loop
PR #31 fixes the accept-error backoff in the capsule-cli proxy to use a host-provided sleep rather than a direct OS sleep call, making the backoff path safe for WASM targets. Without this change, the backoff could block or panic in a WASM runtime that does not expose native threading primitives. The fix is surgical, touching only the error-recovery path in the accept loop so normal throughput is unaffected. This unblocks capsule-cli deployment in browser and edge WASM environments.

Sphere15 PRs merged
Payment-request memo and delivery envelope correctness
PRs #554 and #547 fix the recipient-ECDH payment-request path so that the memo correctly carries the requester nametag and message, and the delivery envelope carries the sender nametag alongside the memo. PR #557 then ensures the full incoming payment-request view is rebuilt on reload rather than relying on potentially stale cached state (fixing issue #556). PR #553 is resolved by the memo fix, and the corresponding sphere bumps (#368, #363) promote the corrected sdk.dev.12 and dev.10 snapshots into the integration branch. Collectively these changes close a data-fidelity gap in the ECDH-encrypted payment-request flow that caused memo and nametag fields to be lost in transit.
sphere-sdk, sphere
Payment-request session convergence across payer devices
PR #53 fixes the wallet-api so that payment-request resolution correctly converges to all of the payer's other active sessions, referencing the behaviour specified in protocol sections §9 and §16. Previously, resolution could terminate at a single session, leaving other devices unaware of the pending payment request. The fix ensures the resolution fan-out is exhaustive before the request is considered settled. This is critical for multi-device wallet users where the paying device may not be the one that originally received the request.
wallet-api
Multi-device vault sync and anti-rollback hardening
PR #544 corrects multi-device sync logic in the vault layer and introduces multi-device-aware anti-rollback guards. Without the fix, concurrent writes from different devices could result in divergent vault states or allow a stale device to roll back to an earlier vault version. The anti-rollback mechanism is now aware of the full device set, so a rollback attempt is rejected if any peer has advanced past the target version. PR #550 complements this by hydrating history from the server on reload and performing a read-back sweep to verify consistency, closing issue #549.
sphere-sdk
SDK build stability, CI reliability, and e2e coverage
PR #561 serialises the tsup build in sphere-sdk to eliminate a race condition that intermittently omitted .d.ts declaration files (issue #548). PR #370 adds per-operation GitHub Pages concurrency controls so that a cleanup job can no longer cancel an in-flight deployment (issue #369). PR #28 in token-api switches sphere-sdk resolution to npm and removes a cross-repo checkout that was causing CI token errors. PR #552 swallows transient PROFILE_NOT_INITIALIZED errors in the profile layer and includes two soak-script fixes. PR #366 extends the two-profile smoke test with a browser-layer reopen family to improve end-to-end coverage of multi-profile flows.

IPFS sidecar rewrites eliminate Python memory leaks
PRs #16 and #18 apply two successive mitigations to memory growth in the nostr-pinner: first reducing httpx and JSON fragmentation pressure, then adding an nginx cache layer with pre-serialised responses to cut allocations further. PR #20 delivers the definitive fix by replacing the Python routing-get, DHT refresh, and WebSocket handlers with a Go sidecar, eliminating the root-cause memory leak entirely. The Go sidecar is a long-running process managed alongside the Python service, keeping the operational model simple while gaining Go's deterministic GC. This sequence of PRs moves the nostr-pinner from leak mitigation to a structurally sound architecture.
ipfs-storage

PR #66 simplifies the ML inference pool, reducing complexity in how inference workers are allocated and managed. Together these PRs complete the rebranding and leave the inference layer in a cleaner state ahead of the hosted-alpha release.
