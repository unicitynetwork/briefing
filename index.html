Astrid42 PRs merged
Self-updating CLI with staged install awareness
PR #905 introduces an in-place, staged, install-aware self-update mechanism to the Astrid CLI, allowing the binary to update itself without disrupting active sessions. The implementation distinguishes between staged and direct installs to apply the correct update path. This eliminates manual upgrade steps and lays the groundwork for reliable operator tooling at scale.
astrid
Reliable multi-subscriber fan-out message delivery
PR #903 fixes a long-standing reliability gap in Astrid's pub-sub layer by introducing concurrent dispatch and loss-free receive for multi-subscriber fan-out. Previously, messages could be dropped or delivered out of order when multiple subscribers were active simultaneously. The fix ensures all registered subscribers receive every message regardless of concurrency, making event-driven capsule communication trustworthy.
astrid
Capsule pipeline migrated to componentized WASM artifacts
Across 19 capsule repositories, CI release pipelines were updated (e.g. capsule-cli #30, capsule-react #24, capsule-identity #25) to publish componentized .capsule artifacts instead of raw WASM blobs. This shift adopts the WASM component model, enabling proper interface typing, composability, and versioned distribution through the capsule registry. All affected capsules also received a dependency bump to astrid-sdk 0.7.1 to align with the updated SDK surface before the new publish format landed.

Sphere73 PRs merged
Offline-resilient boot and reliable fan-out delivery
PR #543 makes Sphere initialization offline-resilient so that a relay connection failure no longer bricks the boot sequence, allowing the SDK to start in a degraded-but-functional state. PR #542 adds fan-out delivery via courier deposit combined with a Nostr blast, with recipient-side deduplication preventing duplicate processing. Together these changes harden the delivery pipeline against both connectivity failures at startup and duplicate message paths in multi-transport scenarios.
sphere-sdk
Vault multi-device sync, anti-rollback, and remote storage wiring
A sustained sequence of vault PRs (#520 through #541) built and hardened RemoteTokenStorageProvider end to end: Phase 6 (#520) added CAS flush, delete-resurrect, pagination, and anti-rollback; Phase 7 (#525) wired provider gates and load state machines; Phases 8.1–8.3 (#526, #527) added real HTTP fetch clients and in-process token-api e2e tests. PR #537 wired RemoteTokenStorageProvider into Sphere as an opt-in backup and sync provider, and #541 stabilized multi-device sync with real-time receive. On the token-api side, PRs #14–#15 surfaced ackNonce, sentSeq, and epochSig on the wire protocol (WIRE-1 through WIRE-3), and #24 added storage-port conformance tests with concurrent idempotency and change-stream re-scan on oplog overflow.
sphere-sdk, token-api
Token API production hardening: auth, durability, and abuse controls
A focused hardening sprint across token-api addressed critical production gaps: #17 enforced MongoDB majority write-concern durability and fixed trustProxy and topology configuration; #18 added backups with PITR support along with a production Dockerfile and CI pipeline, both flagged as BLOCKERs. PRs #21–#23 added body limits, timeouts, a 5xx error handler, health and readiness endpoints, metrics auth, CORS controls, JWT algorithm pinning, nonce expiry, and Sentry removal. PR #22 introduced per-route abuse caps including rate-limited auth routes, per-owner blob quotas, blob GC, recipient scoping, and WebSocket caps. Auth was further stabilized in #26 and #27 by fixing session rotation on re-login to eliminate E11000 duplicate-key errors and replacing rotating refresh tokens with a stateless requireAuth pattern.
token-api
Sphere wallet reliability: reload fix, inventory retry, and fail-closed degradation

Unicity Network6 PRs merged
Testnet2 infrastructure hardening and RDS secret management
PRs #10 (bft-ansible) and #3 (bft-terraform) together migrate the testnet2 gateway RDS password to a self-managed non-rotating secret, disabling the previous 7-day automatic rotation that was causing connectivity disruptions. The Ansible playbook was updated to point the gateway database secret reference at the new self-managed secret. This change provides stable, predictable database credentials for the testnet2 environment while operational processes for secret handling are formalized.
bft-ansible, bft-terraform
Aggregator observability and BFT disk-backed SMT
PR #44 in aggregator-subscription fixes a silent failure mode where 5xx exceptions escaping the handle() function were not recorded in metrics or alerts, meaning database outages were invisible to on-call operators. PR #156 in aggregator-go adds a RocksDB-backed disk SMT (Sparse Merkle Tree) for BFT shard mode, replacing what was presumably an in-memory structure and enabling the shard to survive restarts with persistent state. Together these changes improve both the observability and durability characteristics of the aggregator layer.
aggregator-subscription, aggregator-go
Fulcrum Docker auto-recovery and network ID in mint transactions
PR #17 in Fulcrum-Alpha adds automatic recovery from failed-reorg database inconsistency in the Docker deployment, preventing the node from getting stuck after a reorg event, and also enables WSS serving on port 443 for standard TLS WebSocket connectivity. PR #123 in state-transition-sdk-js adds the network ID field to the Unicity ID mint transaction, addressing issue #122 and ensuring transactions are explicitly scoped to their target network. This prevents cross-network transaction replay and aligns the SDK's mint transaction format with the broader Unicity protocol specification.
Fulcrum-Alpha, state-transition-sdk-js
