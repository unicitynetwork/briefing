Astrid34 PRs merged
Streaming HTTP Airlock and Daemon Extraction
PRs #509 and #507 in astrid introduced a streaming HTTP airlock for capsule communication and refactored the daemon and build tooling into standalone binaries with full lifecycle command support. PR #4 in sdk-rust added the underlying streaming HTTP API that powers this capability. PR #513 followed with boot pipeline fixes and new CLI features to stabilise the new architecture. Together these changes lay the foundation for a more modular, process-oriented capsule runtime.
astrid, sdk-rust
SDK 0.3.0 Release and Ecosystem-Wide Propagation
sdk-rust PR #8 cut the 0.3.0 release, preceded by PR #5 releasing 0.2.2 and PR #6 adding a CI release workflow. All 16 downstream capsule repositories then merged coordinated bump PRs to adopt astrid-sdk 0.3.0, with several (capsule-http, capsule-prompt-builder, capsule-memory, capsule-hook-bridge, capsule-context-engine) also including CI fixes and WASM compatibility patches for uuid. This wave of dependency updates ensures the entire capsule ecosystem is aligned on a single SDK version. The CI release workflow addition means future SDK releases can be automated and consistently reproducible.
sdk-rust, capsule-cli, capsule-skills, capsule-shell, capsule-session, capsule-router, capsule-registry, capsule-react, capsule-prompt-builder, capsule-openai-compat, capsule-memory, capsule-identity, capsule-http, capsule-hook-bridge, capsule-fs, capsule-context-engine, capsule-anthropic
Tool Descriptions and Capsule-Level Documentation
Six capsule repositories merged PR #2 each, adding tool descriptions and capsule-level documentation to their manifests, making tool capabilities machine-readable and discoverable by agents. sdk-rust PR #7 underpins this by introducing support for doc comments as descriptions and an inline mutable flag in the macro system. capsule-skills PR #4 followed with a fix to remove redundant duplicate tool entries from the manifest that arose from this work. These changes directly improve the quality of context provided to LLMs consuming the OpenAI-compatible and Anthropic interfaces.
capsule-openai-compat, capsule-identity, capsule-skills, capsule-shell, capsule-http, capsule-fs, sdk-rust
Boot Pipeline Fixes and Stream Subscription Cleanup
astrid PR #513 addressed multiple boot pipeline regressions introduced during the daemon extraction refactor and extended CLI functionality. capsule-cli PR #3 removed the now-obsolete agent.v1.stream.delta subscription, cleaning up a protocol artifact that is no longer needed given the new streaming HTTP airlock. capsule-openai-compat PR #1 added placeholder hints and updated the default model to improve out-of-the-box usability. These fixes collectively stabilise the end-to-end developer experience after the significant architectural changes landed earlier in the day.
astrid, capsule-cli, capsule-openai-compat

Sphere3 PRs merged
Identity-Aware Connect Protocol for dApps
sphere-sdk PR #92 introduced an identity-aware connect protocol to the SDK's connect layer, enabling dApps to establish sessions that are bound to a specific user identity rather than just a wallet address. This is a foundational change that allows downstream applications to make trust decisions based on verified identity context. The work aligns sphere-sdk with the broader Unicity identity primitives being developed across the ecosystem.
sphere-sdk
Wallet Switch and Logout Notifications with UI Polish
sphere PR #284 added event notifications that inform connected dApps when a user switches wallets or logs out, closing a correctness gap where dApps could operate on stale session state. The same PR also introduced a video background poster to improve perceived load performance in the UI. These changes improve both the reliability of the dApp integration surface and the visual quality of the Sphere shell.
sphere
On-Chain Achievement Sync for Sphere Quest
sphere-quest PR #6 implemented on-chain achievement synchronisation, enabling quest completions and milestones earned within Sphere to be written to and read from the chain. This bridges the off-chain quest experience with verifiable on-chain state, making achievements portable and trustless. The feature is a significant step toward a fully on-chain reputation and engagement layer within the Sphere ecosystem.
sphere-quest
Unicity Network3 PRs merged
Remove Receipt from Certification Request and Response
state-transition-sdk PR #101 and aggregator-go PR #128 jointly removed the receipt field from both the certification request and response structures, addressing issue #92. This simplifies the certification protocol by eliminating an unnecessary data element that was being carried across the request/response boundary. The change requires coordinated updates on both the SDK and the aggregator sides to maintain wire compatibility. Streamlining the certification message format reduces payload size and clarifies the protocol contract for downstream implementors.
state-transition-sdk, aggregator-go
Relay Metrics Instrumentation
unicity-relay PR #4 added metrics collection to the relay service, providing observability into relay performance and throughput. Instrumenting the relay allows operators to monitor message rates, latencies, and error conditions in production environments. This is an important operational capability as the network scales and relay infrastructure becomes more critical to overall system health.
unicity-relay
