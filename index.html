Remove legacy v1 API and storage compatibility layer
PR #147 removes dead v1 API endpoints and associated storage compatibility code from the aggregator-go service. This cleanup eliminates technical debt that had accumulated around the original v1 API surface, along with any shim or translation logic required to bridge old storage formats. Dropping this dead code reduces the maintenance burden on the aggregator and shrinks its attack surface. The change signals that all consumers have fully migrated away from v1, allowing the codebase to move forward without backward-compatibility constraints.

@martti007 owns or co-owns at least 10 of the 16 open items across [state-transition-sdk-java], [state-transition-sdk-js], and [.github]. Any blockage, delay, or context-switching by this one person directly stalls the majority of the release.

Three BFT sharding items ([.github] platform-level, [state-transition-sdk-java], [state-transition-sdk-js]) are all in Todo with no dev activity. BFT sharding is architecturally significant and cannot be rushed through dev, test, and integration in one day. These items should be cut or deferred.
Java SDK V2 tasks blocked by in-dev prerequisites
[state-transition-sdk-java] V2 support for payments/splits and general structure/verification are in test, but V2 Unicity ID support and type tag/proof updates are still in dev or todo. If the in-dev items do not close today, the test items cannot be signed off and V2 Java SDK ships incomplete.
JS SDK TokenId validation bug still in Todo
[state-transition-sdk-js] non-32-byte TokenId acceptance is a correctness bug assigned to @martti007 and has not entered dev. Shipping with this defect risks invalid token state on-chain at TGE, which is a protocol-level integrity issue.
Unicity data structure versioning not started
