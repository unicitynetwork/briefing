Astrid2 PRs merged
Async HTTP client migration in CLI
PR #652 replaces the blocking reqwest client with its async counterpart in the Astrid CLI. This eliminates the dedicated OS thread that reqwest::blocking spins up per request, reducing resource overhead and aligning the CLI's HTTP layer with the rest of the async runtime. The change prevents potential deadlocks that can occur when blocking calls are made inside an async context, such as within a Tokio executor. Overall this makes the CLI more consistent, performant, and safer under concurrent workloads.
unicity-astrid
Sandbox mount ordering and AppArmor resilience fix
PR #651 addresses two distinct but related hardening issues in the workspace sandboxing layer. First, it corrects the bwrap (Bubblewrap) mount ordering, ensuring bind mounts and overlay mounts are applied in the sequence the kernel expects, preventing failed or mis-scoped sandbox environments. Second, it introduces an AppArmor fallback so that systems where the AppArmor profile is unavailable or not loaded do not hard-fail on workspace startup. Together these changes make sandbox initialisation more robust across heterogeneous deployment environments, particularly CI runners and developer machines with varying kernel security configurations.

Sphere1 PRs merged
Reduce failure cooldown from 24 hours to 1 hour
PR #1 in sphere-api shortens the cooldown period applied after a failure event from 24 hours down to 1 hour. This is the first merged PR in the unicity-sphere organisation and introduces the cooldown feature baseline for the API. The previous 24-hour window was overly punitive for transient failures, significantly delaying recovery and degrading developer and user experience. Reducing it to 1 hour strikes a better balance between rate-limiting abusive or broken clients and allowing legitimate retries in a reasonable timeframe. This change will need to be monitored post-deployment to validate that the shorter window does not increase load on downstream services.

Unicity Network1 PRs merged
Unicity ID support in state transition SDK
PR #102 implements Unicity ID support in the JavaScript state transition SDK, closing issue #98. This adds the data structures, serialisation logic, and likely validation routines needed to represent and process Unicity identifiers within state transition flows. Unicity IDs are a foundational primitive for addressing and routing within the network, so their presence in the JS SDK unblocks client-side applications that need to construct or verify state transitions referencing specific network entities. The change extends the SDK's type surface and will require downstream consumers to update any state transition payloads to include the new ID fields where applicable.
