Astrid39 PRs merged
Dual-version WIT runtime restores SDK 0.7.x compatibility
PRs #1108, #20, and #62 form a coordinated breaking-change reversal: process@1.0.0 was restored as a frozen interface while file-injection capabilities were promoted to the new process@1.1.0 interface. Astrid now serves both versions concurrently, re-establishing compatibility with capsules built against sdk-0.7.x. The Rust SDK was re-synced to the restored 1.0.0 WIT and the host import moved to 1.1.0. PR #21 in wit additionally documents all three oversized-cap error codes introduced in process@1.1.0, closing a documentation gap that would have hampered capsule authors hitting the new limits.
astrid, wit, sdk-rust
Per-principal provisioning, isolation, and audit chain
PR #1091 fixed a critical regression where shared runtimes were colliding across principals; the fix loads exactly one runtime instance per content hash, restoring per-principal isolation without spawning duplicate runtimes. PR #1096 extended the CLI so the global --principal flag is honoured during astrid init, enabling per-principal provisioning workflows end-to-end. PR #1062 landed per-action filesystem, network, and process host calls on the signed audit chain, making every sensitive host interaction cryptographically accountable. PR #24 in capsule-registry deduplicated discovered provider entries, preventing phantom duplicates from appearing in provisioning inventories.
astrid, capsule-registry
Gateway reliability: keepalive timeouts and boot budget
PR #1093 introduced a keepalive-driven inactivity timeout on gateway-to-kernel IPC requests, returning typed 504 errors on timeout rather than hanging indefinitely, significantly improving observability and client resilience under load. PR #26 in sage raised the astrid-up boot budget to approximately 22 seconds and added a progress heartbeat so slow-starting environments no longer trip the boot watchdog prematurely. Together these changes harden the runtime boundary between the gateway and kernel layers for production deployments.
astrid, sage
0.9.x / 0.2.x release wave across capsule fleet
Astrid 0.9.0 and 0.9.1 were cut (PRs #1104, #1110) alongside a coordinated 0.2.0 release of the entire capsule fleet covering capsule-skills, capsule-hook-bridge, capsule-openai-compat, capsule-openai, capsule-identity, capsule-session, capsule-shell, capsule-registry, capsule-react, and capsule-cli. Sage 0.2.0 and 0.2.1 and Astralis 0.2.0 and 0.2.1 followed, with floor pins raised to astrid >=0.9.1 to track the new baseline. Astralis received a fix (PR #10) giving openai_temperature an empty-string default so headless init no longer blocks on missing config. CI hygiene fixes landed in homebrew-tap (PR #8) and capsule-http (PR #20), and the sage MCP consent model was corrected in PR #27 to converge grant-on-use consent with durable recorded decisions.
astrid, sage, astralis, capsule-react, capsule-skills, capsule-hook-bridge, capsule-openai-compat, capsule-openai, capsule-identity, capsule-session, capsule-shell, capsule-registry, capsule-cli, homebrew-tap, capsule-http

Critical security hardening: SSRF, RBAC, and token redaction
PR #41 closed an in-org privilege escalation path on member-role endpoints, preventing authenticated members from promoting their own permissions. PR #45 addressed a cluster of high and medium security findings from audit #31: SSRF redirect protection, strict encryption key enforcement, token log redaction to prevent credential leakage, environment-variable secret handling, and fail-closed behaviour on API key validation. These changes collectively eliminate the most severe attack surface identified in the audit and bring the API into compliance with the documented security model.
sphere-api
Operational resilience: graceful shutdown, health checks, cron locking
PR #46 introduced graceful shutdown draining, a Dockerfile HEALTHCHECK instruction, and process guards to ensure in-flight requests complete cleanly before the container exits, addressing audit findings M5 through M7. PR #44 applied a leader-lock to the auto-approve cron sweep so only one replica executes the sweep at a time across horizontally scaled deployments, eliminating the race condition flagged as C2. Together these changes make sphere-api safe to operate under rolling deploys and multi-replica Kubernetes scheduling.
sphere-api
Atomic quest awards and social-platform classification consistency
PR #40 rewrote quest and achievement award logic to be atomic and idempotent, closing a double-credit race where concurrent requests could grant the same reward twice. PR #47 centralised social-platform classification into a single source of truth, eliminating the LINK classification drift that had caused inconsistent behaviour documented as issue #31 L1. Both fixes address correctness bugs that directly affected user-facing reward integrity and platform integration reliability.
sphere-api
CI gating and team onboarding
PR #43 introduced a CI gate that blocks Docker image builds unless lint, typecheck, and unit tests all pass, closing the audit finding H2 that allowed broken images to be published. PR #2 in the design repo adds a coworker, expanding the contributing team. The CI gate is a foundational quality control measure ensuring that only verified builds reach the container registry going forward.
sphere-api, design
