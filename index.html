Restore tool visibility on OpenAI-compatible path
PR #19 in capsule-prompt-builder fixes tool visibility on the OpenAI-compatible path by restoring describe-ACL handling and invalidating a stale cache that was suppressing tool discovery. Pin toolchain and ship SDK 0.7.1 release
PR #900 pins the Rust toolchain to 1.95.0 in CI so that cross-compiled release binaries build reliably, eliminating environment-dependent failures in the release pipeline. PR #54 in sdk-rust cuts the 0.7.1 release, publishing the accumulated fixes under a stable version tag. Sphere27 PRs merged
Bootstrap wallet-api service: scaffold through auth and sessions
PRs #2 through #14 lay the complete foundation for the new wallet-api service, starting with workflow conventions (#2), pre-flight spec alignment (#4), open-decision resolution on STATE_ID_EXISTS aggregator lag (#6), sphere-sdk dev-line pinning (#8), guardrail seed files (#10), and the M0 scaffold and test harness (#12). M1 (#14) delivers auth and session management as the first functional milestone. This sequence establishes the development contract and unblocks all subsequent milestone work.
wallet-api
wallet-api milestones: inventory, mailbox, history, and payment requests
M2 (#18) adds inventory management, a blob store, and intent handling, while M3 (#22) delivers the full mailbox flow including validated deposit, all-status listing, claim handoff, and rejection. M4 (#26) extends the API with client-asserted history, gap-free payment-request sequencing, and pg-boss-based expiry, rounding out the core user-facing capabilities. Companion SDK bumps (#16, #20) keep the wallet-api aligned with sphere-sdk dev.4 and remove a now-redundant dts shim. PR #24 reverts mock proof-snapshotting to avoid a split-resume gap introduced by sphere-sdk#501, with the gap documented for tracking.
wallet-api, sphere-sdk
sphere-sdk thin-wallet core, engine, encryption, and delivery wiring
PR #497 introduces the thin-wallet core: a WalletApiClient (S1), lazy storage port and provider (S2), field-level encryption (S6), and a contract suite (S7). PR #493 adds the recoverable engine (Part E) with deterministic realization and status-agnostic idempotent submission backed by conflict detection. PR #502 completes the wiring by adding a DeliveryProvider port, WalletApiMailboxProvider, PaymentsModule rewiring, and composition presets (S3, S4, S7). PR #499 fixes a publish-pack race that caused dts files to be missing from dev.2 and dev.3 tarballs, ensuring consumers receive complete type information.
sphere-sdk
token-api service bootstrapped with auth, epochs, and SDK signing
PRs #1 through #5 build the token-api from the ground up: A0 (#1) scaffolds a Fastify 5 / Mongoose 9 service with a replica-set test harness and Zod configuration. A1 and A1b (#2) wire dependency injection, establish StoragePort as the single source of truth, and add MongoStorage. A1.5 (#3) introduces SDK-backed test helpers (keys.ts and actions.ts), and A2 (#4) makes the server reuse SDK signing with shared golden test vectors covering cases C1–C3. A3 and A4 (#5) deliver the full auth surface—challenge, verify, refresh, and logout—plus an epoch service, completing the authentication layer.
token-api
soak fixes: chain-pubkey regex, nametag rebind, send args
PR #482 resolves three issues surfaced during soak testing (tracked as sphere-sdk#474): a faulty chain-pubkey regex that caused validation failures, a live nametag rebind that was not being applied correctly, and incorrect send argument handling. These are correctness fixes that improve reliability under sustained load and are a prerequisite for stable soak runs going forward.

Optional burnStateMask and tolerant certification parsing
PR #125 makes burnStateMask optional in TokenSplit.split, removing a breaking requirement that forced callers to supply the mask even when it was not needed. The same PR adds tolerant parsing of CertificationResponse so that unexpected or missing fields no longer cause hard failures. This change unblocks sphere-sdk's burnStateMask wiring (sphere-sdk#495) and improves forward compatibility as the certification format evolves.
state-transition-sdk-js
Disk-backed RocksDB SMT with HA for BFT-shard mode
PR #160 replaces the in-memory sparse Merkle tree in the aggregator with a RocksDB-backed implementation, providing durable on-disk state that survives restarts. High-availability support is added specifically for BFT-shard mode, enabling the aggregator to operate reliably in multi-node deployments. This is a significant infrastructure step toward production-grade aggregator operation and a prerequisite for sustained shard scalability.
