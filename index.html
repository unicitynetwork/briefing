Astrid2 PRs merged
Async HTTP client replaces blocking reqwest in CLI
PR #652 migrates the CLI from reqwest::blocking to the async reqwest client, aligning the command-line layer with the async runtime used elsewhere in the stack. Blocking HTTP calls were a latency and threading concern, as they occupies a dedicated OS thread per request rather than yielding to the async executor. Switching to async reqwest allows the CLI to participate in the Tokio event loop cleanly, improving responsiveness and resource efficiency. This is a correctness and performance fix with no functional behaviour change for end users.
unicity-astrid
Bubblewrap mount ordering and AppArmor fallback hardened
PR #651 addresses two distinct sandbox reliability issues in the workspace isolation layer. First, the bwrap (Bubblewrap) mount sequence was reordered to ensure bind mounts are applied in a deterministic order, preventing race conditions or failed mounts that could silently break the sandbox environment. Second, an AppArmor fallback path was introduced so that systems without a loaded AppArmor profile do not hard-fail but instead degrade gracefully. Together these changes make workspace sandboxing more robust across heterogeneous Linux host configurations.

Sphere1 PRs merged
Fail cooldown slashed from 24 hours to 1 hour
PR #1 reduces the cooldown period applied after a failure event in the sphere-api from 24 hours down to 1 hour. This is the first merged PR in the unicity-sphere organisation, establishing the initial baseline for cooldown policy. A 24-hour penalty was overly punitive for transient or recoverable failures, blocking legitimate retries and degrading overall system availability. Shortening the window to 1 hour strikes a better balance between back-pressure and recovery agility, allowing participants to re-engage far sooner after resolving an issue.

Unicity Network1 PRs merged
Unicity ID support added to state-transition SDK
PR #102 implements Unicity ID support in the JavaScript state-transition SDK, closing issue #98. Unicity IDs are the canonical identity primitive across the network, and their absence from the JS SDK was a gap preventing client applications from constructing or validating identity-bearing state transitions natively. The change adds the necessary data structures, serialisation logic, and any required validation to handle Unicity IDs end-to-end within the SDK. This brings the JS SDK to parity with other language bindings and unblocks downstream application developers building on the state-transition layer.
