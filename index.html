Astrid7 PRs merged
Async per-capsule WASM runtime with instance pooling
PR #828 lays the foundational runtime overhaul enabling async execution of individual WASM capsules backed by an instance pool, establishing the architectural groundwork called out by the #821–#826 series. Each capsule now runs in its own isolated async context, eliminating head-of-line blocking across concurrent principal workloads. Instance pooling reduces cold-start overhead by reusing pre-warmed WASM module instances. Per-principal isolation ensures one principal's execution environment cannot interfere with another's, a critical correctness and security property for multi-tenant capsule hosting.
unicity-astrid
Per-principal CPU fuel ledger and budget enforcement
PRs #821 and #823 introduce a shared per-principal CPU fuel ledger that aggregates consumption across all capsules belonging to a principal, then enforce a hard CPU-rate budget ceiling against that ledger. PR #821 wires the shared fuel accounting so that every capsule debits the same principal-scoped counter rather than tracking in isolation. PR #823 adds the enforcement layer that halts or throttles execution once the principal's CPU budget is exhausted. Together these two PRs make CPU fairness a first-class runtime guarantee rather than a best-effort heuristic.
unicity-astrid
Live resource-usage read-path and admin contract
PR #822 defines the kernel-level admin contract for per-principal resource usage, introducing the UsageGet message type and the ResourceUsage data structure as the canonical API surface. PR #824 populates the read-path with live data, replacing any placeholder stubs so that queries reflect real-time per-principal CPU consumption. PR #825 exposes this data over the HTTP gateway via the new GET /api/sys/principals/:id/usage endpoint and corrects a CPU value fix in the QuotasView component. This chain turns raw fuel accounting into observable, queryable operational telemetry.
unicity-astrid
CLI quota visibility with CPU ceiling reporting
PR #826 extends the Astrid CLI with the 'astrid quota show' subcommand, surfacing each principal's current usage against its configured budget alongside a visual CPU ceiling line. This gives operators and developers an immediate, human-readable signal of how close a principal is to its rate limit without needing to query raw API endpoints. The ceiling line provides at-a-glance context that a bare number cannot, making capacity planning and incident triage faster. This completes the observability loop opened by the kernel and gateway work in #822–#825.

Sphere6 PRs merged
Token-engine foundation with frozen ITokenEngine contract
PR #398 (Phase 0) introduces the token-engine subsystem by establishing the foundational scaffolding and freezing the ITokenEngine interface as the authoritative contract for state-transition v2. Freezing the interface at this stage prevents API drift as downstream phases build on top of it, giving all consumers a stable integration target. The PR defines the core state-transition abstractions that subsequent token-engine phases will implement. This is a deliberate phase-gated approach that prioritises contract stability over feature completeness in the initial merge.
sphere-sdk
CID delivery pipeline re-enabled with raised inline cap
PR #395 re-enables automated CID delivery in the payments flow and raises the inline payload cap from its previous limit to 512 KiB, reducing the frequency with which content must be externalised to IPFS. PR #31 completes the wiring on the CLI side by connecting publishToIpfs and cidFetchGateways inside buildSphereProviders, ensuring the SDK's delivery logic is reachable from CLI-initiated payment flows. Together these two PRs close the gap opened by sphere-sdk#394, making end-to-end CID-backed payment delivery functional again. The raised cap materially improves throughput for medium-sized assets that previously required an unnecessary round-trip.
sphere-sdk, sphere-cli
Payment guard and SENT reconciliation bug fixes
PR #392 addresses two distinct correctness issues in the payments subsystem: a missing guard that now checks sourceTokenIds before proceeding, and a load-tail reconciliation bug where SENT transactions were not being properly accounted for at the tail of a loaded payment sequence. PR #388 is a broader V6-RECOVER patch resolving defects #387, #389, and #390 simultaneously, introducing durable invalid-state handling and a root-cause reconstruction mechanism for source-state corruption. The source-state reconstruction is particularly important because it allows the system to recover a consistent view of payment history without manual intervention. These fixes harden the payments layer against edge cases that could previously result in silent data inconsistency.
sphere-sdk
Presigned URL fix: sign asset-id as AMZ metadata header
PR #30 corrects a presigned URL generation bug where the x-amz-meta-asset-id value was being included in the request body or query string rather than signed as a header, causing signature validation failures on upload. Signing the header explicitly ensures AWS S3 rejects any request where the asset-id has been tampered with in transit, restoring the integrity guarantee the presign mechanism is supposed to provide. This is a targeted fix with no API surface change, but it unblocks media upload flows that were failing signature verification in production-like environments.
sphere-api

CORS and API key expiration fixes for aggregator-subscription
PR #36 enables CORS on the /health endpoint, which was previously rejecting cross-origin probes from browser-based monitoring and dashboard tooling. PR #35 fixes a V1 API key expiration bug where keys were not being correctly invalidated or their expiry was being misread, potentially allowing stale credentials to remain active. Both fixes are operational correctness patches that improve the reliability and security posture of the aggregator-subscription service. Unblocking cross-origin health checks in particular enables a broader class of uptime monitoring integrations without requiring server-side proxy workarounds.
aggregator-subscription
Migrate minting to networkId and salt token ID scheme
PR #119 implements the migration described in issue #115, replacing the previous pattern where callers supplied their own token IDs at mint time with a deterministic scheme based on networkId combined with a salt. Caller-supplied IDs introduced a collision risk and made token provenance harder to audit, since any caller could choose an arbitrary identifier. The networkId-plus-salt approach ties each minted token's identity to the network context and a unique entropy value, making IDs both collision-resistant and traceable. This is a breaking change to the minting API surface and aligns the JS SDK with the canonical token identity model used elsewhere in the Unicity stack.
