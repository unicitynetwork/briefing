Astrid2 PRs merged
WASM runtime upgraded to wasmtime Component Model
PR #640 migrates the WASM execution engine in the capsule subsystem from Extism to the wasmtime Component Model. This is a significant runtime change that aligns Astrid with the emerging WASM Component Model standard, enabling stronger interface typing via WIT and improved cross-language interoperability. The move away from Extism reduces a third-party abstraction layer, giving the team more direct control over guest module lifecycle and host function bindings. This lays a more stable foundation for capsule execution as the Component Model spec matures toward stabilization.
unicity-astrid
Homebrew tap auto-updates on every release
PR #624 introduces automation to keep the Homebrew tap in sync whenever a new Astrid release is published. Previously, updating the tap required a manual step, creating friction and risk of version drift for macOS users installing via Homebrew. The new workflow triggers on release events and pushes the updated formula automatically, ensuring the tap always reflects the latest published version. This improves the developer experience for local installation and reduces release toil for the maintainer team.

PR #287 adds a dedicated Sphere Connect documentation section to the Sphere site, complete with hash-anchor navigation support. Twitter API request cost optimised per user
PR #21 in the sphere-quest repository introduces optimisations to reduce the cost of Twitter API calls on a per-user basis. Unoptimised API usage against the Twitter v2 tier can incur significant rate-limit pressure and monetary cost at scale, so this change likely batches, caches, or de-duplicates requests to minimise redundant calls. By scoping the optimisation per user, the fix targets the granular consumption unit that drives billing, making spend more predictable as the Quest user base grows. This is a meaningful operational improvement that protects infrastructure budget and improves response efficiency for Twitter-integrated features.
