Astrid26 PRs merged
Purge stale WASI target refs across all capsules
PRs #7, #18, #21, #14, #14, #16, #16, #17 swept every capsule repo to remove obsolete wasi-target references from release workflows. The stale references were leftover artefacts from a prior build target strategy and posed a risk of silently misconfiguring CI pipelines. This housekeeping pass by @joshuajbouw ensures release workflows reflect only current, supported targets and reduces confusion for contributors onboarding to the build system.
capsule-agents, capsule-memory, capsule-identity, capsule-http, capsule-hook-bridge, capsule-fs, capsule-context-engine, capsule-cli
Migrate all capsules to per-domain WIT and wasm32-unknown-unknown
Seventeen breaking-change PRs (#6 through #19 across repos) executed a coordinated migration away from a shared monolithic WIT definition toward per-domain WIT interfaces, simultaneously switching the compile target from wasm32-wasi to wasm32-unknown-unknown. The per-domain WIT split allows each capsule to version and evolve its interface contract independently without coupling to unrelated domains. Adopting wasm32-unknown-unknown removes the WASI runtime dependency, broadening the environments in which capsule Wasm modules can be embedded. The breaking-change designation signals downstream integrators to update host bindings and any tooling that previously expected WASI imports.
capsule-agents, capsule-cli, capsule-context-engine, capsule-fs, capsule-hook-bridge, capsule-http, capsule-identity, capsule-memory, capsule-openai, capsule-openai-compat, capsule-prompt-builder, capsule-react, capsule-registry, capsule-router, capsule-session, capsule-shell, capsule-skills
Refresh Homebrew tap SHAs after 0.7.0 retag
PR #6 updated the SHA checksums in the Homebrew formula following a retag of the 0.7.0 release. A retag invalidates previously recorded asset hashes, causing Homebrew installs to fail checksum verification until the formula is patched. This single-commit chore by @joshuajbouw restores a working brew install path for end-users on macOS.

Sphere16 PRs merged
Profile token migration and provider re-enablement
PRs #289 (sphere-sdk) and #307 (sphere) together deliver a full bidirectional legacy-to-Profile token storage migration and re-enable Profile mode in the UXF build. sphere-sdk #289 implements the two-way migration so tokens written under the legacy storage layout are transparently promoted to Profile-backed storage and can be demoted again, preventing data loss during rollout. sphere #307 gates the Profile provider swap behind this migration path, directly addressing the regression that forced the revert in #306, where swapping providers silently discarded existing tokens. PRs #305 and #306 document the failed first attempt: #305 switched UXF to Profile-backed providers with Vite Node-stub aliases but was immediately reverted via #306 once the missing migration was identified, providing a clear before-and-after in the history.
sphere, sphere-sdk
OpLog oversized-write and CBOR fallback fixes
PRs #287, #288, and #281 address data-integrity problems in the Profile storage layer. #287 and #288 (the latter superseding the former with a corrected scope tag) migrate oversized OpLog writes in CommunicationsModule and GroupChatModule to CID-references, preventing writes that exceed block size limits from corrupting the log. #281 adds a CBOR fallback for SENT-record deserialisation and introduces an aggregator-spent recovery sweep, hardening the recovery flow against records written in older encodings. Together these fixes close the issues tracked as #285 and #280 and make the Profile backend safe for production OpLog volumes.
sphere-sdk
Performance and reliability improvements to Profile transport
PRs #265, #271, #273, #277, and #279 form a series of incremental stability improvements to the Profile and payments subsystems. #265 adds auto-merge for monotonicity violations in the CRDT log and gates peer broadcasts behind an off-by-default flag, preventing runaway replication during early rollout. #273 decouples the per-flush HEAD-verify check from the at-least-once delivery gate, eliminating spurious flush failures under network jitter. #279 bounds helia blockstore fetches with an LRU cache and in-flight deduplication, capping memory growth under concurrent reads. #277 persists Nostr event dedup state across process restarts, and #271 classifies recipient-address-mismatch VerificationErrors as permanent so the payments retry loop does not spin indefinitely on unresolvable failures.
sphere-sdk
Observability, CLI output routing, and SDK vendor bump
PR #276 introduces toggleable verbose debug logging with namespace filtering, span tracking, and field-level redaction, giving operators fine-grained control over diagnostic output without exposing sensitive data in production logs. sphere-cli #29 corrects wallet use confirmation messages to be routed to STDERR rather than STDOUT, ensuring CLI consumers that parse STDOUT are not disrupted by human-readable status text. sphere #304 bumps the vendored sphere-sdk snapshot to the integration/all-fixes branch at cca4530, pulling in the cumulative fixes from PRs #275, #278, #280, and #282 into the main sphere repository. sphere-sdk #283 closes two residual issues from §D: deterministic asset ordering and a round-trip correctness fix for invoice serialisation.
sphere, sphere-sdk, sphere-cli
