Astrid2 PRs merged
Cooperative yielding and WASM getrandom support
Two PRs from @joshuajbouw address runtime reliability and cross-platform build correctness. PR #806 fixes cooperative yielding in the async event loop so that broadcast storms no longer starve other tasks, improving responsiveness under high-throughput event conditions. PR #802 injects the correct getrandom cfg flag for wasm32-unknown-unknown capsule targets, ensuring cryptographic randomness is properly wired when compiling to WebAssembly. Together these changes harden Astrid for both server-side load spikes and browser-based deployments.

Sphere7 PRs merged
Sphere SDK vendor bumps and bundled perf stack
PRs #328 and #333 advance the vendored sphere-sdk snapshot twice across the week, first to 956b79f and then to 1499a93, pulling in a series of incremental improvements. The anchor of the first bump is PR #381, a combined 6-PR performance stack covering CAR batching, Rule-4 provenance tracking, pointer caching, clear suppression, pin retry logic, and a V6-RECOVER ledger entry. The second bump consolidates PRs #383 and #384 into the vendor tree, making their fixes immediately available to Sphere consumers. This pattern of stacking SDK work before vendoring keeps the Sphere repo history clean while delivering compound gains in a single integration step.
unicity-sphere, sphere-sdk
CID reference error degradation and 404 console storm fix
PR #383 fixes a page-freeze regression in the communications layer by degrading CID_REF_UNREADABLE errors gracefully rather than letting them propagate fatally, mirroring the equivalent fix applied to GroupChat in issue #334. PR #384 introduces an adaptive cold-cache strategy for sidecar IPFS profile probes, suppressing the flood of 404 errors that previously littered the console during cache-miss sequences. Together these two SDK PRs reduce both visible error noise and the risk of UI freezes when content identifiers are temporarily or permanently unresolvable.
sphere-sdk
UXF and layout probe accuracy improvements
PR #332 resolves issue #331 by replacing a shallow IndexedDB name probe with content-aware Sphere context probes in the UXF layer, ensuring that context detection reflects actual stored data rather than mere database existence. PR #330 resolves issue #329 by switching the layout aggregator to a direct probe path, bypassing an SDK pinger that was producing false-negative readiness signals. Both fixes address cases where optimistic or superficial checks gave Sphere incorrect confidence about system state, leading to downstream rendering or initialisation failures.

Unicity Network4 PRs merged
SIF Firewall rebrand and dynamic policy engine
PR #36 rebrands the sif-dash interface from its working title to the official product name SIF Firewall, aligning the UI with external messaging. PR #53 delivers a substantial new capability: per-request execution plans, a weighted policy combiner, validation hardening, and accompanying load tests, giving operators fine-grained control over how policies are selected and composed at request time. This dynamic policy system allows SIF to adapt enforcement logic without redeployment, which is critical for environments with heterogeneous or rapidly changing trust requirements.

Audit log implementation and review fixes
PR #50 introduces the initial audit log feature, establishing the infrastructure to record security-relevant events within the SIF Firewall. PR #54 follows up with a set of review-driven fixes, addressing issues identified during code review of the original implementation. A complete and accurate audit trail is essential for compliance and post-incident forensics, so getting the foundational record-keeping correct before further feature work proceeds is a high-priority hardening step.
