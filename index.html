Astrid3 PRs merged
Gateway auth: bearer revocation and invite token API
PR #768 introduced a full HTTP admin API alongside invite token primitives, laying the groundwork for controlled onboarding flows through the gateway. PR #775 built on this by adding bearer token revocation triggered on principal deletion, ensuring credentials are invalidated immediately when an identity is removed. Together these two PRs close the lifecycle loop for authentication tokens in the gateway layer. This makes the gateway significantly more production-ready from a security standpoint.
unicity-astrid
Fix CORS origin wiring in gateway router
PR #774 corrected a misconfiguration where the cors_allow_origins setting was not being threaded through into the actual router, meaning CORS policy was silently not enforced. The fix ensures that configured origin allowlists are now correctly applied at the routing level. This is a critical correctness fix for any deployment where the gateway is exposed to browser-based clients.

Sphere16 PRs merged
Resilient profile storage: IDB, pinning, and snapshot loading
PR #331 replaced the in-memory blockstore with an IDBBlockstore for browser-persistent profile data, added a pointer-publish gate to prevent premature publishing, and introduced a token fallback path. PR #317 added OpLog block pinning to prevent garbage collection of critical blocks and requested persistent storage from the browser, emitting a critical-block-evicted event when eviction occurs anyway. PR #314 implemented lazy loading of profiles from a local snapshot blob with background remote sync, dramatically improving cold-start latency. PR #309 hardened identity-key fallback storage and added tolerance for LoadBlockFailedError so that transient storage failures no longer crash the profile load path. Collectively these changes make profile persistence robust against browser storage pressure and intermittent connectivity.
sphere-sdk
Connectivity surface, offline gating, and BLOCKED state recovery
PR #315 introduced a sphere.connectivity surface to the SDK, providing a structured API for querying network state, and added send-path gating so operations are held or rejected gracefully when offline. PR #327 softened the connectivity gate from a hard block to an advisory, preventing overly aggressive rejection of operations under degraded but functional network conditions. PR #320 added logic to automatically clear the BLOCKED profile state when a successful pointer poll is observed, handling cases where the block was set due to transient connectivity loss rather than a genuine error. These changes give applications a principled way to respond to network conditions without sacrificing data integrity.
sphere-sdk
OpLog epoch reset and local development infrastructure
PR #316 introduced an OpLog epoch-reset primitive with a permanent walkback floor, allowing the operation log to be safely reset to a prior epoch without risking loss of data below a defined minimum boundary. PR #322 wrapped local infrastructure images for the relay, faucet, and aggregator services with ssl-manager, enabling HTTPS in local development environments and closing the gap between local and production TLS behaviour. These two PRs address operational concerns at opposite ends of the stack — data integrity primitives and local dev fidelity.
sphere-sdk
Service status banner, dev endpoint overrides, and layout fixes
PRs #317 and #318 introduced a service-status banner that probes the aggregator, IPFS, Nostr, and Markets API endpoints and displays live health state to the user, with PR #318 fixing the banner truth logic and forcing a re-probe on re-initialisation. PR #319 corrected the IPFS gateway probe to query the gateway directly rather than routing through the SDK IpfsPinger, improving probe accuracy. PRs #320 and #321 added developer override capabilities for the aggregator URL and then extended those overrides to Nostr, IPFS, Faucet, and Market endpoints via console or localStorage. PR #322 surfaced all these dev endpoint overrides in a Settings modal GUI, and PR #323 fixed a z-index conflict where BaseModal was rendering beneath the MobileBottomNav. These changes together give developers fine-grained control over service routing and give end users visibility into connectivity health.

BFT snapshot support in unicity-node
PR #3 introduced Byzantine Fault Tolerant snapshot functionality to the unicity-node, enabling the node to capture and restore consistent state snapshots that are safe under BFT consensus assumptions. Snapshots are a foundational operational primitive, allowing faster node bootstrapping and recovery without replaying the full history of the chain. This is a significant infrastructure milestone for the network layer.
