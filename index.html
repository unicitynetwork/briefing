Astrid21 PRs merged
LLM Model Selection, Onboarding, and Provider Discovery
PRs #1016, #1011, #1025, and capsule-registry #21 together deliver a full model-selection pipeline: the CLI gains provider and model onboarding selects (#1016), the gateway exposes an HTTP API to list and bind the active LLM model per principal (#1011), and the registry adds model-granular selection, a new `models` CLI verb, and self-healing on provider upgrade (registry #21). On the provider side, capsule-openai #10 enumerates all catalog models as selectable entries instead of hardcoding, while capsule-openai-compat #24 discovers upstream models dynamically via `/v1/models` rather than pinning to an env variable. Documentation in #1025 and openai-compat #22 rounds out the rollout with an updated onboarding guide, corrected `base_url` examples, and a complete `[env]` schema reference.
unicity-astrid, capsule-registry, capsule-openai-compat, capsule-openai
Grant-on-Use Consent Flow Across Gateway, MCP, and Broker
Three tightly coupled PRs implement a full grant-on-use consent lifecycle: #1001 adds the core elicit-and-grant mechanism that intercepts access to an ungranted capsule and prompts for consent rather than silently dropping the request, #1003 wires this into the MCP layer with a shim that elicits consent and automatically re-sends the original request once a grant is issued, and sage #20 surfaces `grant_required` events and relays the grant decision on the broker side for grant-on-use consumers. Together these changes close the silent-drop UX gap and give users a transparent, auditable path to capsule access. The agent loop also gains fast-fail behaviour on unconfigured chat (#1007), preventing confusing hangs when LLM prerequisites are missing.
unicity-astrid, sage
Security Enforcement: Per-Principal Access, SSRF Fixes, and Scoped Tokens
PR #993 introduces per-principal capsule access enforced kernel-side at dispatch, ensuring capsules are only reachable by explicitly authorised principals rather than any authenticated caller. This is underpinned by #996, which refactors the codebase to adopt a decomposed security-enforcement model, separating concerns for easier auditing and future policy extension. On the network boundary, #1021 closes HTTP SSRF airlock bypasses and surfaces `airlock-rejected` errors so callers receive explicit feedback rather than silent failures. PR #999 adds tweakable per-device capability scopes on pair-device tokens, allowing fine-grained restriction of what each paired device is permitted to do. Complementing these, #1026 adds a per-capsule local-egress allowlist specifically for private LLM endpoints, preventing unintended outbound calls from capsules that should only reach internal infrastructure.
unicity-astrid
CLI Ergonomics, GitHub Source Resolution, and Distro Version Enforcement
PR #1010 adds root-level shorthand aliases for capsule CLI verbs, reducing the command depth needed for common operations and improving developer ergonomics. PR #1000 enables the gateway uplink to resolve GitHub capsule sources before install, so operators can reference capsules directly by GitHub URL without a separate fetch step. The distro fix in #1024 enforces the `[distro].astrid-version` field against the running CLI version at `init` time, catching version mismatches early and preventing silent compatibility failures. PR #1006 addresses a compile-time regression by pinning the MCP dependency to exactly rmcp 1.8.0, ensuring reproducible builds across environments.
unicity-astrid

Wallet-API Production Autoscaling and DB Pool Tuning
PR sphere-infra #16 provisions ECS autoscaling for the wallet-api production service with a range of 2–10 tasks, providing elasticity under variable load without manual intervention. Complementing this, sphere-infra #15 sets `DB_POOL_MAX` per environment — 25 for staging and 80 for production — right-sizing database connection pools to match the expected concurrency at each tier; the updated `deploy.sh` also preserves the live image tag during deploys to avoid accidental image rollbacks. Wallet-api #74 adds a dedicated Autoscaling row to the production Grafana dashboard, making scale-out and scale-in events immediately visible alongside existing service metrics. Together these changes harden the production wallet-api for sustained traffic growth and give the on-call team the observability needed to validate autoscaling behaviour in real time.
