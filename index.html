Sphere4 PRs merged
Trader command tree mirrors trader-ctl interface
PR #5 adds a full trader command tree to sphere-cli, deliberately mirroring the existing trader-ctl command structure. This alignment reduces cognitive overhead for operators already familiar with trader-ctl and ensures consistent CLI ergonomics across tooling. The addition expands sphere-cli's coverage of core trading workflows accessible directly from the unified CLI surface.
sphere-cli
Host transport hardening and pre-flight safety floor
PR #4 enforces a minimum timeout floor of MIN_TIMEOUT_MS=100 on the pre-flight --timeout flag for the host command, preventing dangerously low or zero timeout values from being passed at startup. This defensive guard ensures the host DM transport layer has a baseline window to complete handshake and readiness checks. The fix addresses an edge case where user-supplied values below the floor could cause silent failures or misleading error states.
sphere-cli
Legacy bridge and host DM transport phase 2
PR #1 delivers the second phase of sphere-cli, introducing legacy bridge support alongside host DM (direct messaging) transport integration. This phase connects sphere-cli to older infrastructure components via the bridge layer while simultaneously enabling DM-based host communication. The combined delivery means sphere-cli can now serve both modern and legacy deployment topologies from a single binary.
sphere-cli
Integration tests against live public testnet
PR #2 introduces an integration test suite that runs against real public testnet infrastructure rather than mocked or local environments. Testing against live endpoints validates that sphere-cli command flows, transport handshakes, and timeout behaviours hold under genuine network conditions. This raises confidence in the pre-flight checks introduced in PR #4 and the DM transport work from PR #1, as regressions will surface against actual node behaviour.

Unicity Network3 PRs merged
BFT sharding support in aggregator layer
PR #146 introduces Byzantine Fault Tolerant sharding into the aggregator-go service, enabling the aggregation layer to partition and process state transitions across shards with BFT safety guarantees. This is a significant architectural addition that aligns aggregator-go with the sharded topology being formalised across the SDK layer. Distributing aggregation load across BFT shards improves both throughput scalability and fault isolation in the presence of malicious or faulty nodes.
aggregator-go
StateId shard validation and updated CBOR tags
PR #111 adds a shard membership check on StateId, ensuring that state identifiers are validated against their expected shard before processing, closing issue #107. PR #109 updates CBOR serialisation tags for both the aggregator layer and token types, resolving issue #103 and aligning the JS SDK's wire format with current protocol expectations. Together these two changes tighten correctness at the SDK boundary: shard mismatches are caught early and serialised payloads exchanged with aggregator-go will now use the canonical tag values. Both fixes are prerequisite hygiene for the BFT sharding work landing in aggregator-go via PR #146.
