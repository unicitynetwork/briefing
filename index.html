Payments and Accounting Correctness Fixes
PRs #128, #129, and #130 address a cluster of correctness issues in the payments and accounting subsystems. PR #130 ensures un-finalized tokens remain in a 'submitted' status rather than advancing prematurely, preventing false-positive settlement signals. PR #129 fixes a migration gap in importInvoice, ensuring tokenInvoiceMap is properly carried over during invoice imports. PR #128 exposes getTokenIdsForInvoice on the dependency facade so downstream consumers in the swap flow can reliably resolve token associations.
sphere-sdk
Market Resilience and Payment Mode Passthrough
PRs #131 and #132 extend the reliability and configurability of market and payment operations. PR #132 introduces a retry mechanism paired with a circuit breaker pattern for transient API failures in the market module, reducing cascading failure risk under intermittent connectivity. PR #131 exposes a transferMode passthrough on the payInvoice call in the accounting layer, giving callers explicit control over transfer semantics without requiring workarounds. Together these changes harden the SDK against real-world network unreliability and expand invoicing flexibility.
sphere-sdk
SDK Exports Cleanup and IPFS Deprecation
PRs #150 and #135 address housekeeping and forward-looking maintenance in the SDK. PR #150 fixes broken exports introduced by PR #145, restoring correct module resolution, and adds the recoverPubkeyFromSignature utility to the public API surface. PR #135 soft-deprecates IpfsStorageProvider along the IPNS path, signalling its eventual removal while maintaining backward compatibility for existing consumers. These changes keep the SDK surface clean and guide integrators away from infrastructure dependencies that are being phased out.
sphere-sdk
Referral Nametag Exposure in API and Frontend
PRs #21 and #8 deliver end-to-end inclusion of nametag data in the referrals experience. PR #21 extends the GET /me/referrals API response to include the nametag field alongside existing referral data, removing the need for clients to perform a separate lookup. PR #8 fixes nametag display in the sphere-quest-frontend, ensuring the newly available field renders correctly in the UI. Together these changes give users clear, human-readable identity context within the referrals flow.
