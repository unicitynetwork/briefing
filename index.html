Sphere2 PRs merged
Sphere CLI bootstrap and pointer namespace wiring
PRs #10 and #11 together establish the foundational CLI experience for sphere-cli. PR #10 introduces a top-level init command, a dedicated pointer namespace, and an opt-in flag for surfacing mnemonics in non-TTY environments, giving operators a structured entry point for node bootstrapping. PR #11 closes a correctness gap where the pointer flush was skipping anchor publication on wallets with zero balance, ensuring the anchor is always committed regardless of wallet state. Together these changes make the CLI usable end-to-end for initial deployment and pointer lifecycle management.

Unicity Network6 PRs merged
Java SDK state transition and proof verification overhaul
Five consecutive PRs by @martti007 substantially harden the state-transition-sdk-java. PR #59 updates inclusion proof verification logic and introduces explicit CBOR tags for typed encoding. PR #62 adds shard ID verification cross-checked against the state ID, preventing misrouted state transitions from being accepted silently. PR #63 adds full unicity ID support to the SDK data model, and PR #65 enforces stricter CBOR parsing, splits proof deduplication into a discrete step, and adds an issuer check against the unicity ID — closing the loop on provenance validation. Collectively these changes tighten the trust boundary around state proofs and prepare the SDK for production-grade identity-aware token flows.
state-transition-sdk-java
Predicate-based addressing and mint justification restructure
PR #61 replaces raw address references with predicate objects throughout the SDK, aligning the data model with a capability-oriented ownership scheme. As part of the same change, mint justification is moved to the mint root rather than being carried inline, which simplifies proof construction and makes the justification auditable at the root level. This refactor affects core token lifecycle methods and is a prerequisite for the unicity ID issuer checks introduced in PR #65.
state-transition-sdk-java
NIP-33 DELETE cliff fix for hot relay pubkeys
PR #20 by @MastaP resolves a bug in the unicity-tokens-relay where NIP-33 parameterised-replaceable event DELETE operations were hitting a processing cliff when the originating pubkey was classified as hot. The fix moves the deletion driving logic to the tag side rather than relying on pubkey-side resolution, ensuring DELETE plans are evaluated and applied correctly under high-throughput key conditions. This matters for relay correctness because missed DELETEs would leave stale replaceable events visible to downstream consumers.
unicity-tokens-relay
