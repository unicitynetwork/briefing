Connect SDK: Session Persistence and Wallet Lock Handling
PRs #1 (unicity-claude-marketplace), #6 (sphere-sdk-connect-example), and #101 (sphere-sdk) form a coordinated documentation push by @KruGoL covering three layers of the Connect plugin stack. The updates document session resume flows, allowing clients to reconnect without re-initiating a full handshake, and add explicit handling for the LOCKED wallet state so integrators can respond gracefully when the user's wallet is inaccessible. PR #101 also introduces safeSend, a wrapper that guards against sending messages on a connection that is not yet ready. Together these changes give SDK consumers a clear, end-to-end reference for building resilient Connect integrations.
unicity-claude-marketplace, sphere-sdk-connect-example, sphere-sdk
Nostr DM Reliability and App Reload Fixes
@MastaP merged three fixes addressing runtime stability in the Sphere app and its underlying SDK. PR #288 in sphere resolves an app reload regression, restoring correct state after the application is refreshed or restarted. PR #289 targets Nostr direct message bugs, correcting message delivery or ordering issues in the DM flow. On the SDK side, PR #99 in sphere-sdk ensures the client re-subscribes to Nostr relay events after a reconnect, preventing a silent loss of incoming events that would occur when a relay connection dropped and recovered.
sphere, sphere-sdk
Sphere Quest Day-1 Fixes and Admin Project Creation
Two PRs from @KruGoL address early issues in the Sphere Quest feature. PR #24 bundles a set of day-one fixes identified after initial rollout, stabilising the core quest experience. PR #23 tightens the admin project creation flow by making the project description field mandatory and ensuring the projectId is correctly passed through when a project is created, preventing downstream references from receiving a null or undefined identifier. These changes improve data integrity and reduce the risk of broken quest configurations reaching production.

PostgreSQL Conflict Retry and JSON Validation Fixes
PR #12 in unicity-relay by @MastaP adds automatic retry logic for serializable transactions that fail due to PostgreSQL serialization conflicts, a common occurrence under concurrent write load when using the SERIALIZABLE isolation level. Without this retry the relay would surface transaction errors to callers rather than resolving them transparently, so the fix improves reliability under contention. PR #4 in unicity-node by @lploom corrects an invalid JSON issue, likely a malformed response or payload that could cause parse failures in consuming services. Both fixes harden the network layer's data handling and reduce the surface area for intermittent failures in production.
unicity-relay, unicity-node
