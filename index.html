Async HTTP Migration and Sandbox Hardening
Two fixes from @joshuajbouw address runtime correctness and sandboxing reliability in the Astrid CLI and workspace layers. PR #652 replaces the blocking reqwest client with its async counterpart, eliminating potential thread-pool starvation in async runtimes and aligning the CLI with the project's async-first architecture. PR #651 resolves bwrap mount ordering issues that could cause sandbox initialisation failures, and adds an AppArmor fallback to ensure confinement is maintained on systems where the primary profile is unavailable. Together these changes improve both the reliability of outbound HTTP calls and the robustness of the sandboxed execution environment.

Sphere1 PRs merged
Reduce Failure Cooldown for Faster Recovery
PR #1 from @igmahl shortens the fail cooldown window in the Sphere API from 24 hours to 1 hour. This change dramatically reduces the penalty period following a failed operation, allowing clients and services to retry significantly sooner without manual intervention. The original 24-hour window was overly conservative for typical transient failure scenarios and was impeding development velocity and production recovery times. Tightening the cooldown to 1 hour strikes a better balance between abuse prevention and operational responsiveness.

Unicity Network1 PRs merged
Unicity ID Support in State Transition SDK
PR #102 from @martti007 implements Unicity ID support in the JavaScript State Transition SDK, closing issue #98. This addition integrates first-class identity primitives into the SDK, enabling consumers to construct and validate state transitions that are anchored to Unicity-native identifiers. The change extends the SDK's core data model and likely touches serialisation and validation logic to handle the new ID format correctly. Supporting Unicity IDs at the SDK level is a foundational step for applications that need deterministic, network-verifiable identity references within state transition workflows.
