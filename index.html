Astrid24 PRs merged
WebAssembly-first kernel and runtime overhaul
PRs #1150, #1143, #1141, #1134, and #1132 form the core wasm32-unknown-unknown porting effort. The kernel was refactored to compile for wasm32 (#1150) by inverting native resource acquisition out of Kernel::new (#1137), and a new astrid-runtime crate (#1141) was introduced as a target-selected task/timer facade that presents a unified async API regardless of whether the host is native or wasm. The Wasmtime engine was target-gated inside astrid-capsule (#1143), and a dedicated wasm-clean astrid-capsule-types crate was extracted (#1134) so type definitions carry no host dependencies. Together these changes mean Astrid's kernel semantics layer compiles and links cleanly on wasm32, unlocking browser and edge deployment targets.
astrid-kernel, astrid-capsule-types, astrid-runtime
Async storage surfaces unblock wasm32 boot path
PRs #1155 and #1153 address the final boot-blocking issues on wasm32 by making the audit storage surface (#1155) and CapabilityStore persistence (#1153) genuinely async. Both carried breaking-change markers because their trait signatures changed from synchronous to async, requiring callers to await persistence operations. On wasm32 there is no thread-blocking primitive, so synchronous storage calls would deadlock or panic at runtime; the refactor replaces those calls with properly polled futures. With these two fixes the kernel now boots end-to-end on wasm32-unknown-unknown without panics or hangs.
astrid-kernel, astrid-audit
Capsule reliability: proxy safety and consent correctness
PR #1149 bounds framed socket writes on the CLI proxy so a stalled uplink client can no longer freeze the entire process. PR #1151 injects AstridHome directly into EventDispatcher rather than resolving it from the process environment, eliminating a class of misconfiguration bugs in containerised or wasm contexts. PR #1122 tightens consent by scoping grant-on-use to matching capsules and fixing shim grant resolution looping, while #1119 wires the watch uplink correctly so tools/list_changed notifications are actually delivered over MCP. Together these close several reliability gaps that would surface in production multi-capsule deployments.
astrid-capsule, astrid-kernel, astrid-cli-mcp
CLI tooling: check command, daemon hygiene, update reporting
PR #1130 ships astrid capsule check, a static CI linter that validates tool wiring without running the daemon, giving pipelines a fast gate on capsule configuration correctness. PR #1128 adds a load-time warning when an advertised tool has no execute route, catching misconfiguration early. PR #1139 detaches the spawned daemon into its own session so it survives terminal closure cleanly, and #1125 ensures the stop command confirms actual process exit and reaps upgrade-swapped wedge processes. PR #1126 fixes update --check to report availability across every configured install method rather than stopping at the first match.
astrid-cli, astrid-daemon

Sphere15 PRs merged
Durable split-checkpoint resume for payment intents
PRs #636, #638, #639, #89, and #90 implement sphere-sdk#501 option b end-to-end: a SplitCheckpointStore is ported into the engine (#636), a WalletApiCheckpointStore is added on the API side (#638), and the intent_progress record durably persists the engine-checkpoint burn-proof (#89) so that a split operation can resume from exactly where it stalled after any restart. The seed-holder gate (#90) ensures only the rightful holder can close a checkpointed intent at its terminal step, preventing unauthorised finalisation. A live crash-resume drill was un-scoped and targeted specifically at the split checkpoint path (#639) to validate the full round-trip under simulated failure. This work eliminates the data-loss window that previously existed when a split operation was interrupted mid-flight.
sphere-sdk, wallet-api
Intent observability and operational resilience
PR #84 adds Retry-After headers on 429 RATE_LIMITED responses so clients know exactly when to retry, complementing #632 which adds retry-with-jitter, Retry-After handling, and quieter pump logging on the SDK client side. PR #92 introduces a stuck-split gauge and seed-gate reject counter so operators have metric signals when intents stall or are rejected. PR #633 and the matching sphere bump (#401) keep intents in OPEN state when an engine operation may have certified, avoiding premature failure on possibly-certified sends (#631). PR #86 de-noises the CRITICAL missing-object alert and ensures the scheduled audit fires exactly once per invocation (#85), reducing alert fatigue.
wallet-api, sphere-sdk, sphere
Agentic chatbot: LLM flexibility and DM safety controls
PR #13 makes the LLM backend env-driven and migrates kbbot to ArliAI using an OpenAI-compatible interface, allowing operators to swap models without code changes. PR #14 adds a DM blocklist and per-sender rate limiting to the sphere-bot, protecting the bot from abuse and ensuring fair resource allocation across senders. These two PRs together make the agentic chatbot layer both more configurable for deployment operators and more robust against malicious or runaway users.
agentic-chatbot
Infrastructure: RDS right-sizing after relay purge
PR #26 downsizes the Nostr relay RDS instance from a larger class to db.m7g.large following the kind-31113 event purge. The purge removed the dominant source of write and storage load, making the previous instance size unnecessarily expensive. This is a cost-optimisation change with no functional impact on relay behaviour.
