Enumerate-Capabilities API Lands Across Full Stack
PRs #869 (astrid), #55 (sdk-rust), #16 (sdk-js), and #13 (wit) deliver end-to-end support for a new enumerate-capabilities system call. The WIT interface gains the enumerate-capabilities entry in astrid:sys@1.0.0, the host runtime de-stubs the capability check and implements the syscall, and both the Rust and JS/TS SDKs expose capabilities::enumerate / capabilities.enumerate surface. This gives capsule authors a first-class, portable way to introspect which capabilities are available at runtime rather than relying on trial-and-error or hard-coded assumptions.
astrid, sdk-js, sdk-rust, wit
Persistent-Process Tier Registry and SDK Surface
The persistent-process tier is now a complete, cross-layer feature via PRs #867 (astrid), #53 (sdk-rust), #14 (sdk-js), and #12 (wit). The WIT definition extends astrid:process@1.0.0 with the persistent process tier contract, the host implements the corresponding registry, and both SDKs expose the new process management surface. Persistent processes allow capsules to maintain long-lived worker state across invocations, unlocking stateful agent patterns that were previously impossible within the ephemeral capsule model.
astrid, sdk-js, sdk-rust, wit
IPC Manifest Overhaul: Drop Interceptors, Add Subscribe Tables
PR #863 is the core breaking change: [[interceptor]] and the ipc_* arrays are removed from the manifest format, with tables now being the sole IPC-intent surface and a new [subscribe] table adding priority semantics. Capsule manifests for capsule-agents (#9), capsule-openai (#6), capsule-system (#9), and capsule-cli (#23) are all migrated to the new [subscribe] / [publish] table format in lockstep. The capsule-cli proxy additionally gains forwarding for admin and correlated request topics that were previously dropped. This rationalises the manifest schema to a single, consistent declaration surface and eliminates the ambiguity between interceptor and subscription semantics.
astrid, capsule-agents, capsule-openai, capsule-system, capsule-cli
Capsule Runtime Hardening: Memory, Pooling, Sandboxing, Dispatcher
A cluster of runtime reliability PRs lands together: #840 introduces a per-principal peak-memory ledger with a WASM ResourceLimiter, #839 replaces the fixed 16-slot instance pool with a dynamic host-sized pool, and #838 splits the host semaphore into separate blocking and async-IO gates to reduce contention. PR #857 fixes a regression where the macOS sandbox was silently disabled on macOS 15+, and #841 prevents closed per-topic consumers from being handed back to callers, eliminating a burst-stall condition. Supporting capsule fixes include capsule-react #22 resolving a TurnState KV visibility race in orchestration responses, capsule-context-engine #19 bounding compaction hook fan-out to avoid unbounded waits, and capsule-registry #17 re-discovering providers on cold-principal model lookups. Collectively these changes make the capsule runtime significantly more predictable under load and on modern macOS hosts.
astrid, capsule-react, capsule-context-engine, capsule-registry

v2 Token Engine: Adapter, Factory, and Full Injection
Track A's core deliverable this period is a production-grade v2 token engine, built across PRs #407 (ITokenEngine adapter, factory, and identity), #409 (edge-case hardening), #411 (tokenId, on-chain memo, and data-token surface for Q1/Q2/Q3), and #418 (construction and injection of the v2 engine into Sphere using networkId from the trust base). PR #420 extends the work by persisting v2 token blobs through the storage build/parse pipeline, and #421 vendors the DIRECT:// derivation path to be fully v1-free with five golden-locked test vectors. Injecting the v2 engine on all address paths is confirmed by audit fixes in #425, and a live testnet2 end-to-end wallet payment path validates the full stack in #426. This track eliminates all remaining v1 engine dependencies and establishes a stable, auditable token layer for the network.
sphere-sdk
Accounting Layer: Bulk Returns, Null-Sender Refunds, Invoice Pipeline
PR #405 adds returnAllInvoicePayments as a bulk companion to the existing single-invoice return, and sphere-cli #39 exposes this via a new invoice return command supporting both bulk and nametag targets. PR #400 routes invoice delivery through the token pipeline rather than a bypass path, closing a correctness gap. PR #413 fixes null-sender back-direction refunds by introducing a destinationAddress fallback for attribution, with #414 cleaning up the now-unnecessary workaround from #404. PR #402 wires publishUxfBundle into the OUTBOX/SENT accounting path and adds a recovery-exhaustion event. Together these changes make the accounting layer complete and consistent for the full invoice lifecycle including edge cases around refunds and delivery failures.
sphere-sdk, sphere-cli
State Transition SDK Migration (Track B)
A parallel migration track driven by @igmahl lands across PRs #408, #410, #412, and #416, progressively migrating state transition logic into the SDK. These PRs represent an iterative refactor that moves state machine ownership from application-layer code into a canonical SDK surface, improving reusability and testability. The incremental merge strategy allows continuous integration against the main branch while the larger structural change is assembled in slices.
sphere-sdk
CLI UX Polish: Canonical Flags, Nametag Support, Help and Fixes
