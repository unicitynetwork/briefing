Sphere13 PRs merged
SSL Docker Deployment with HAProxy and CI Hardening
PRs #291, #298, #299, and #300 collectively delivered and hardened a production-grade SSL Docker deployment stack for Sphere. #291 introduced the foundational HAProxy integration and SSL-terminating Docker entrypoint, while #298 tightened the entrypoint logic and added CI validation for Dockerfile.ssl. #299 re-enabled the Dockerfile.ssl path in the validate matrix after it was inadvertently dropped, and #300 completed a second-round hardening pass covering drain budget tuning, CI smoke test fixes, and trimming of the SSL_REQUIRED flag. Together these changes bring Sphere's deployment infrastructure to a hardened, continuously validated SSL-first posture.
unicity-sphere
Resilient Transport Layer with Retry and Health Checks
PRs #108 and #113 overhauled the SDK's transport layer to be significantly more fault-tolerant. #108 introduced resilient event publishing, an active health-check mechanism, and a shared address resolver to reduce redundant lookups across modules. #113 built on this by adding automatic retry logic when a relay rejects a publishEvent call, preventing silent message loss under transient relay failures. These changes are critical for maintaining reliable token and payment event delivery in adversarial or degraded network conditions.
unicity-sphere, sphere-sdk
Concurrent Send Race Condition Eliminated in SpendQueue
PRs #87 and #107 addressed a long-standing TOCTOU (time-of-check/time-of-use) race condition in the payments layer. #87 identified and fixed the core concurrency bug in concurrent send() calls where overlapping state reads could result in double-spends or dropped transactions. #107 followed up by introducing the SpendQueue abstraction, which serialises outbound payment operations to structurally prevent the race from recurring rather than relying on ad-hoc locking. This significantly improves payment reliability under high-concurrency wallet usage.
sphere-sdk
AccountingModule, SwapModule, and CLI Overhaul
PRs #109, #110, and #111 delivered three major feature additions to the SDK surface. #109 introduced AccountingModule, providing a full invoice lifecycle including creation, tracking, and payment attribution against received tokens. #110 added SwapModule, implementing a P2P token swap protocol backed by an on-chain escrow mechanism for trustless exchange between parties. #111 completed a comprehensive CLI overhaul adding structured help output, shell completion support, and dedicated subcommands for the new accounting and swap capabilities, making the full feature set accessible from the command line. SDK versions 0.6.14 and 0.7.0 (sphere PRs #294 and #296) propagated these features into the Sphere application.

Unicity Network1 PRs merged
Sharded v2 Inclusion Proof Composition in Aggregator
PR #142 implemented sharded inclusion proof composition for the v2 proof format in the Go aggregator. The change introduces logic to aggregate inclusion proofs across shards, enabling the aggregator to produce a single coherent v2 proof from data distributed across multiple shard state trees. This is a foundational scalability improvement, as sharding allows the aggregator to partition its workload horizontally without sacrificing the ability to generate verifiable cross-shard proofs. Correct proof composition is essential for client-side verification and trustless auditability of the Unicity network state.
