Unicity Network8 PRs merged · 7 new branches
SSL Manager Alias Proxy Stabilised & Hardened
A rapid sequence of six ssl-manager PRs (#6 through #11) progressively built up and then stabilised a new nginx alias proxy mechanism. Work began with #9 refactoring alias registration to operate at the app HTTPS port and removing the old TLS proxy startup, followed by #8 adding a 10-second TLS handshake timeout to prevent thread exhaustion caused by health check floods. Crash resilience was then addressed in two passes: #7 introduced a bash supervisor wrapper for SIGSEGV recovery, and #6 added automatic restart on any crash. PR #10 generalised the approach by having ssl-setup auto-generate nginx alias blocks, and #11 fixed a final gap by ensuring the alias proxy correctly forwards to the primary HTTPS endpoint for full API access. The matching branches (fix/alias-proxy-auto-restart, fix/alias-proxy-shell-supervisor, fix/alias-proxy-handshake-timeout, refactor/nginx-alias-instead-of-proxy, feat/generic-nginx-alias-blocks) track each incremental step and are all now merged.
ssl-manager
IPFS Storage Migrated to SSL Manager Nginx Aliases
Two ipfs-storage PRs (#4 and #5) migrated the service away from its own TLS proxy setup toward the centralised nginx alias blocks now provided by ssl-manager. PR #4 (branch refactor/nginx-alias-blocks) replaced the existing TLS proxy server blocks with nginx alias server blocks, aligning ipfs-storage with the new architecture. PR #5 (branch refactor/use-ssl-manager-alias-nginx) completed the transition by refactoring to consume the generic alias nginx config generated by ssl-manager rather than maintaining any local variant. This reduces duplication and ensures ipfs-storage benefits automatically from any future ssl-manager improvements. The two PRs together make the storage layer a clean consumer of the shared SSL infrastructure.
ipfs-storage
