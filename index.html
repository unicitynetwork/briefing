Astrid7 PRs merged
IPC pub/sub ACL gains subtree wildcard routing
PR #883 introduces trailing-asterisk wildcard semantics to the publish/subscribe ACL, so a single rule now matches an entire subtree of IPC addresses rather than requiring exhaustive enumeration. PR #25 in capsule-cli immediately leverages this by correcting the proxy ACL: it drops the crashing admin-response subscribe entries and opens the correct MCP front-door paths. Together these changes make ACL configuration both more expressive and less error-prone for capsule operators.
unicity-astrid, capsule-cli
Guest crash logs and principal resolution surfaced to host
PR #885 fixes a silent failure mode in the daemon run-loop where ERROR-level logs emitted inside a guest capsule were swallowed and never forwarded to the host daemon log, making crashes invisible to operators. PR #878 corrects the ipc::recv path so that the invoking principal's profile is properly resolved rather than arriving as a null or incorrect identity. PR #56 in sdk-rust ensures that the tool_describe macro publishes its descriptor into the fan-out bus, which is a prerequisite for correct describe responses across multi-tool capsules. These three fixes collectively close a class of silent, hard-to-diagnose runtime failures.
unicity-astrid, sdk-rust
Astrid 0.8.0 release and CI hygiene
PR #854 cuts the 0.8.0 release, consolidating the IPC ACL, guest-log surfacing, and principal-resolution fixes into a stable tag. PR #889 standardises the PR template by renaming the 'Test Plan' section to 'Verification', aligning terminology across the monorepo and reducing ambiguity about what evidence is expected before merge.

V2 engine cutover: v1 SDK removed, token minted at registration
PR #480 performs the breaking removal of the v1 state-transition-sdk, making the v2 engine the sole execution path and eliminating the dual-stack maintenance burden. PR #481 adds mint-and-store logic so that a UnicityIdToken is created and persisted at nametag registration time, anchoring identity to the v2 token model from the outset. PRs #345 and #346 in the sphere app repo merge the full sdk-migration-v2 feature branch, and PR #344 bumps the app to sphere-sdk 0.8.0-dev.6 to pick up per-network storage isolation. PR #484 refreshes all SDK documentation to reflect v2 semantics, ensuring the public API surface is accurately described after the cutover.
sphere-sdk, sphere
Per-network storage isolation for token DB, KV, and IPNS
PR #472 partitions the token database, key-value store, and IPNS state by network using the chainPubkey as the namespace identifier, preventing data bleed when a client operates against multiple networks. PR #485 extends this isolation to the PENDING_V2_DELIVERIES journal, which had remained unscoped and could mix delivery state across networks. PR #344 in the sphere app bumps to the dev.6 build that includes both fixes, making the isolation end-to-end. This is a prerequisite for any multi-network or testnet/mainnet coexistence scenario.
sphere-sdk, sphere
Payments, market pricing, and transfer verification hardened
PR #479 adds verification of incoming v2 transfers and makes the validate() call engine-aware, so that payment proofs are checked against v2 rules rather than silently passing through legacy logic. PR #483 switches all price fields in the market module to decimal-string bigints, eliminating floating-point precision loss on large token amounts. PR #468 hardcodes and exports DEFAULT_ESCROW_ADDRESS as @escrow-test-02, giving the swap module a stable, well-known escrow anchor for testing. These changes collectively close correctness gaps in the financial flow before broader trader rollout.
sphere-sdk
Trader CLI wrapper, local faucet, and soak-test playbook
PR #49 in sphere-cli adds spawn/stop lifecycle commands for the sphere trader process alongside local-* host primitives, giving operators a first-class CLI interface for trader management. PR #45 replaces the HTTP faucet dependency with local token minting, removing an external network requirement from the test setup. PR #475 delivers a roundtrip soak test, a demo playbook, a spec-drift audit against the current implementation, and investigation notes from issue #473, providing both regression coverage and a reproducible demo path. PR #6 in sphere-infra documents per-project GitHub Actions deploy policies, formalising the IAM model for automated deployments.
