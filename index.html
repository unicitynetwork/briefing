Capsule component fleet promoted to 0.2.x

capsule-react, capsule-agents, capsule-system, capsule-router, capsule-prompt-builder, capsule-memory, capsule-fs, capsule-context-engine, astralis
MCP daemon session stability and self-contained Codex plugin
PR #1177 fixed a regression in the CLI where MCP daemon sessions were not kept alive, which would cause tool integrations relying on long-running MCP processes to silently drop. oracles #4 made the Codex MCP plugin fully self-contained, removing external runtime dependencies that previously required the host environment to supply them. Together these two fixes harden the MCP integration layer that underpins AI tooling within Astrid OS. The changes are particularly important ahead of broader developer adoption following the 0.9.4 launch.
astrid, oracles
Oracles dependency hygiene via Dependabot
Three automated Dependabot PRs landed against the oracles repository: #3 bumped the sha2 Rust crate from 0.10.9 to 0.11.0, bringing cryptographic primitive alignment with the current RustCrypto release line. #2 advanced the actions/checkout GitHub Action from v4 to v7, and #1 updated softprops/action-gh-release from v2 to v3, keeping CI and release automation on supported action versions. These routine bumps reduce exposure to known vulnerabilities and ensure the oracles CI pipeline uses well-maintained tooling.

Sphere10 PRs merged
Oracle API re-keying without full reinitialisation
sphere-sdk #659 introduced the new setOracleApiKey method, allowing operators to rotate the live token-engine API key at runtime without tearing down and reinitialising the entire oracle connection. Complementing this, sphere #420 closed a keyless L3-send window by gating all sends until the oracle key is confirmed present, preventing transactions from being dispatched in an unauthenticated state. sphere #418 refined AGGREGATOR_API_KEY handling so the key is required only when subscriptions are disabled and is intentionally ignored when they are active, eliminating a class of misconfiguration errors. These three PRs collectively tighten the security boundary around the token engine's key lifecycle.
sphere-sdk, sphere
Per-wallet aggregator subscriptions and quota system
sphere #414 introduced a per-wallet aggregator subscription model with an accompanying quota system, enabling fine-grained control over aggregator access on a wallet-by-wallet basis rather than globally. sphere #415 extended this with a Docker-oriented runtime-swappable subscription configuration and a same-origin /sgw reverse-proxy, making it straightforward to reconfigure subscription behaviour in containerised deployments without a rebuild. sphere #417 addressed review findings from the initial subscription PRs, adding network pinning, dormancy gating, and proper poll-cancellation logic to prevent stale polling loops. wallet-api #97 bumped the sphere-sdk dependency from 0.9.1-dev.5 to 0.11.5 so the wallet API consumes the production-ready subscription and oracle key APIs.
sphere, wallet-api
Node 24 LTS and npm 12 publish pipeline fix
sphere-sdk #660 upgraded both the publish workflow and general CI to Node 24 LTS after npm 12.0.0, which ships with the new sigstore signing integration, was found to be incompatible with older Node runtimes. sphere-sdk #661 then removed the forced npm@latest pin from the publish step entirely, because npm 12.0.0 itself contained a broken sigstore implementation that caused publish failures regardless of Node version. The two-PR sequence first establishes the correct runtime baseline and then eliminates the unstable toolchain pin, restoring a reliable automated publish path for the sphere-sdk package.
sphere-sdk
Wallet token validation as opaque container
wallet-api #95 removed explicit mint-reason checking from token validation, treating the token as an opaque container and validating only its structural integrity rather than asserting anything about why it was minted. This decouples the wallet API from application-level minting semantics, making it more robust against future changes to mint-reason conventions upstream. The change aligns with the broader subscription and oracle key hardening work in this cycle, ensuring the validation layer does not inadvertently gate valid tokens based on metadata that is outside its trust boundary.
wallet-api, sphere

Semantic detection-rule tuning and over-refusal guard
semanticd #75 delivered a rules-only update to the bm01 detection ruleset, tuning existing detection rules and introducing an over-refusal guard to prevent the semantic daemon from incorrectly rejecting legitimate inputs. Over-refusal in a detection context degrades user experience and erodes trust in the system's decisions, so the guard establishes a balance between precision and recall. The bm01 rules-only scope means no model weights were changed, making the update fast to deploy and easy to roll back if edge cases surface in production.
semanticd
SMT updated to version 6a in state-transition SDK
state-transition-sdk-js #130 resolved issue #129 by upgrading the Sparse Merkle Tree implementation from version 3o to version 6a. Version 6a brings structural improvements to the SMT that affect proof generation and verification within the JavaScript state-transition SDK. Keeping the SMT version current is critical for interoperability with other components of the Unicity Network stack that produce or consume SMT proofs, and falling behind on SMT versions can introduce subtle proof-verification incompatibilities across SDK consumers.
