Sphere7 PRs merged
Transaction history lazy-loading and infinite spinner fix
PR #393 introduced lazy-loading for the transaction history list, deferring off-screen item rendering to improve initial paint performance on accounts with large histories. PR #394 fixed a consequent regression where the lazy-load spinner entered an infinite spin state; the root cause was an Intersection Observer attached to the wrong scroll ancestor, which was corrected by rooting the observer at the actual scroll container. Together these two PRs (#393, #394) deliver a performant and correctly terminating history list. PR #395 bumps sphere-sdk to 0.10.7 to pull in the corresponding SDK-side changes.
sphere
Deferred send 'delivery pending' status surfaced to users
PR #392 surfaces a 'delivery pending' state in the send flow for transactions that complete with a deferred delivery outcome (labelled P4 priority). PR #391 pairs this with a sphere-sdk bump to 0.10.6 and wires the new transfer:delivery_pending event type into the application event handling layer. This gives users explicit feedback rather than ambiguous silence when a send is accepted but not yet confirmed, reducing support burden around uncertain transaction states.
sphere
Wallet-API retry logic for transient network failures
PR #627 adds automatic retry behaviour for idempotent GET requests in the wallet-api client when a transient NETWORK_FAILURE is detected. Retrying only idempotent methods ensures safety, avoiding duplicate side effects on non-idempotent calls. This improves resilience in flaky or mobile network conditions without requiring callers to implement their own retry logic. The change is contained to sphere-sdk, making it available to all consumers of the wallet-api surface.
sphere-sdk
sphere-sdk v2 README and wallet-api setup docs rewrite
PR #628 rewrites the sphere-sdk README and its referenced documentation in preparation for the v2 API surface. The stated motivation is closing a wallet-api setup gap that builders were consistently hitting, suggesting the previous docs omitted or obscured a critical onboarding step. Updated documentation at this stage ensures external developers integrating against v2 have accurate guidance before a formal release is cut. Clear setup docs reduce friction for ecosystem builders and decrease inbound support load on the core team.
sphere-sdk
