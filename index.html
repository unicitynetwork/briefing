Astrid5 PRs merged
Meta Harness activation on exact host prompt turns
PRs #56 (aos-ce) and #26 (oracles) together wire up the Meta Harness to fire precisely on exact host prompt turns rather than on broader scheduling intervals. PR #56 introduces the trigger condition inside aos-ce so the harness activates only when the host prompt turn count matches the configured threshold. PR #26 complements this by routing proactive host hooks directly to the exact agent sessions that issued them, preventing misdirected hook delivery across concurrent sessions. Together these changes tighten determinism in agent orchestration and eliminate spurious harness invocations that could corrupt session state.
aos-ce, oracles
Telegram capsule decoupled and rebranded as Unicity AOS
PRs #51 (aos-ce) and #21 (capsule-telegram) cleanly separate the Telegram capsule from the core aos-ce runtime. PR #51 removes the vendored copy of the Telegram capsule that was previously bundled inside aos-ce, reducing the core dependency surface and making capsule versioning independent. PR #21 updates the capsule-telegram documentation to present the capsule under the Unicity AOS identity rather than as a generic runtime artifact, aligning public-facing materials with the project's branding. This decoupling allows the Telegram capsule to ship on its own release cadence without requiring aos-ce changes.
aos-ce, capsule-telegram
Capsule-telegram migrated to astrid-sdk 0.7.1 and wasm32 ABI
PR #20 upgrades capsule-telegram from an older astrid-sdk version to 0.7.1 and simultaneously migrates the compilation target to the wasm32-unknown-unknown ABI. The wasm32-unknown-unknown target removes the dependency on WASI system interfaces, making the compiled capsule portable across any compliant WebAssembly runtime without OS-level shims. Adopting astrid-sdk 0.7.1 brings in API surface changes required by the newer ABI contract and ensures the capsule can consume updated SDK primitives introduced alongside this release. This migration is a prerequisite for deploying capsule-telegram in sandboxed AOS environments that do not expose a WASI layer.

Unknown token types rejected by default in state-transition SDK
PR #137 in state-transition-sdk-js introduces a strict rejection policy for any token type not explicitly recognised by the SDK. Previously, unrecognised token types could pass through state transition validation silently, creating a potential vector for malformed or malicious token structures to enter the state machine. The change enforces an allowlist approach at the SDK boundary, returning a defined error for unknown types rather than treating them as a no-op. This hardens the JS SDK against future token type proliferation that has not been whitelisted, and ensures downstream integrators receive explicit failures rather than subtle undefined behaviour.
