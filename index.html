Astrid14 PRs merged
Signed AOS release pipeline and capsule publishing
PRs #23 and #25 in aos-ce introduce a full signed AOS release pipeline culminating in the publication of signed Community capsule assets, establishing cryptographic integrity for AOS distribution. The homebrew-tap repository received a parallel hardening series: #1 adds ShellCheck before CI validation, #2 enforces checksum verification of AOS release artifacts, and #4 and #5 fix formula commit and update workflows while requiring verified formula updates. Together these PRs close the end-to-end supply-chain gap between a tagged AOS release and a verifiable Homebrew install.
aos-ce, homebrew-tap
AOS runtime health, version parsing, and state migration
PR #18 introduces a loopback AOS runtime health check, giving the system an internal signal path to verify runtime liveness without external dependencies. PR #21 updates the CI pipeline to fetch runtime releases from the astrid-runtime repository, decoupling runtime artifact sourcing from the main repo. PR #24 fixes robust parsing of the staged AOS version string, preventing fragile pattern matches from breaking downstream pipeline steps. PR #22 hardens product state migration, reducing the risk of data loss or inconsistency when users upgrade between AOS product versions.
aos-ce
Capsule and Audit boundary documentation refresh
PRs #19 and #20 in aos-ce refine how capsules are described, first presenting them as Unicity AOS components and then consolidating that framing into consistent product language across the docs. PR #4 in the .github repository clarifies the Unicity Audit boundary, ensuring that contributors and auditors have a precise definition of what falls within the scope of a formal review. These documentation changes accompany the new release pipeline to ensure the public-facing narrative matches the technical reality of the signed capsule distribution model.

Sphere4 PRs merged
Connect intent delivery states and send semantics docs
PR #435 in sphere surfaces two previously invisible delivery states—delivery-pending and undeliverable—for connect intents and the event bridge, giving users and integrators actionable feedback when a message cannot be immediately confirmed as delivered. PR #672 in sphere-sdk documents intent results and send delivery semantics in depth, and adds Node.js recipient composition examples for the connect API. Together these changes reduce ambiguity around asynchronous message delivery and provide SDK consumers with the reference material needed to handle failure modes correctly.
sphere, sphere-sdk
Unicity ID address derivation prompt and key upgrade
PR #436 adds a prompt for the user's Unicity ID when deriving a new address in Sphere, tying address generation explicitly to the user's identity layer rather than allowing silent derivation. PR #432 delivers a substantial subscription upgrade: in-place key upgrade, acknowledgement-based key delivery, and a real renewal flow replace the previous stub implementations. The ack-based delivery model ensures the client only advances its key state once the counterparty has confirmed receipt, preventing key desynchronisation during network interruptions.
sphere

PRs #18 and #19 in bft-terraform roll the testnet2 gateway image forward twice—first to c99150f-cfg1 referencing aggregator-subscription #65, then to 0e675eb-cfg1 referencing #67—ensuring the live testnet tracks the latest aggregator build and validates the upgrade flow in a real environment.
aggregator-subscription, bft-terraform
Signature recovery byte verification in state transition SDK
PR #136 in state-transition-sdk-js adds explicit verification of the recovery byte when checking signatures, closing a gap where a valid-looking signature with a malformed recovery byte could pass validation. The recovery byte is critical for reconstructing the public key from a compact ECDSA signature; without verifying it, ambiguous signatures could be accepted against the wrong key. This hardening improves the correctness guarantees of state transition proofs submitted to the Unicity network.
state-transition-sdk-js
Website ISR stability and HubSpot form error surfacing
PR #23 adds a 5-minute Incremental Static Regeneration safety net to the Next.js site, preventing deploys from reverting to seed content when the CMS data has not yet propagated at build time. PR #22 surfaces HubSpot validation errors directly on lead forms so that users receive immediate, descriptive feedback when their submission is rejected rather than a silent failure. Both fixes address production-visible regressions and directly improve the reliability and usability of the public Unicity website.
